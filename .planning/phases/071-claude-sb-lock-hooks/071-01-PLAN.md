  <action>
Create new file `hooks/lib/phase-path.sh` mirroring the style of `hooks/lib/nofollow-guard.sh`:

1. First line: `# shellcheck shell=bash` (no shebang — file is sourced, not executed; matches nofollow-guard.sh).
2. Header comment block describing purpose ("path-to-phase resolver, used by phase-lock-claim.sh and the informational peek in completion-audit/stop-check"), CONTEXT.md decision reference, and a Usage section showing both the match and no-match cases.
3. Define `_resolve_phase_from_path()`:
   - Takes one argument: a filesystem path (absolute or relative).
   - Match regex: `\.planning/phases/([0-9]{3})[-/]` (per CONTEXT.md `### Path-to-phase resolution`).
   - On match: print the captured 3-digit phase number to stdout via `printf '%s' "$NNN"` (no trailing newline — callers may compose), then `return 0`.
   - On no match: print nothing, `return 0`. Callers treat empty stdout as "not a phase-locked path" (per CONTEXT.md — the resolver MUST NOT return non-zero on no-match, so hooks can use `phase=$(_resolve_phase_from_path "$path") || true` and proceed).
   - Implementation uses bash's `[[ ... =~ ... ]]` plus `BASH_REMATCH[1]` (no awk/sed dependency, no fork). Save+restore any prior `BASH_REMATCH` is unnecessary because the function defines its own scope when called.
4. Do NOT add `set -euo pipefail` (this is a sourced library, not a script).
5. Do NOT make the file executable (`chmod +x` is wrong for sourced libs — `nofollow-guard.sh` is mode 644).

Implementation reference (CONTEXT.md "Path-to-phase resolution"):
- regex `\.planning/phases/([0-9]{3})[-/]` matches both directory path (`.planning/phases/071-foo/`) and direct file (`.planning/phases/071-foo/bar.md`) and also slash-only-suffix (`.planning/phases/071/`).
- Padded 3-digit output is what `phase-lock.sh` expects per Phase 70 (`_pl_normalize_phase` outputs 3-digit form).

Verify the regex covers the test cases below before declaring done.
