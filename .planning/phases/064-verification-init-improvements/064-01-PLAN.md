## Title: VFY-01 — Intermediate Verification Enforcement: Design Spec

### Problem Statement
The current enforcement model has two tiers:
- Tier 1 (intermediate commits): requires only silver-quality-gates
- Tier 2 (final delivery): requires full required_deploy list including verification-before-completion

However, verification-before-completion is currently only checked at the final delivery gate (gh pr create / deploy). There is no enforcement at intermediate task boundaries — Claude can complete individual tasks within a phase without triggering verification. A user could execute all plans but skip verification-before-completion until the very last commit.

### Design: Intermediate Task Boundary Verification

**Hook event**: PreToolUse/Bash is the natural insertion point. completion-audit.sh already fires on this event for git commit and git push. A new check can be added to completion-audit.sh's Tier 1 path (or a dedicated hook) that fires when a task-boundary signal is detected.

**Task boundary signals** (what triggers the check):
The following command patterns indicate a task is being declared complete:
1. `git commit` — atomic commit marking a task as done within a plan
2. Plan completion signal: a `cat` / write to a `*-SUMMARY.md` file (PostToolUse/Write could fire here)
3. Explicit "task done" invocation: executor calls `/gsd:verify-work` (PostToolUse/Skill)

The most practical signal is `git commit` (already intercepted by completion-audit.sh). This is the natural task boundary in GSD's execution model — each task produces an atomic commit.

**What would block completion at a task boundary**:
For the intermediate verification gate to be meaningful without being disruptive, the recommended design is:
- NOT blocking individual commits (that would break GSD's atomic-commit model)
- BLOCK at the plan-completion boundary: when all tasks in a plan are done and the executor tries to write the *-SUMMARY.md or run the plan's final commit, check that a subset of verification steps are present
- Alternatively: add verification-before-completion to Tier 1 (required_planning) for plans that are explicitly verification plans (plans whose name contains "verify" or "verification")

**Recommended implementation path** (deferred to a future phase):
1. Add a new config key `required_verification` to `.silver-bullet.json` (default: `["verification-before-completion"]`)
2. Extend completion-audit.sh Tier 1 check: when `is_intermediate=true` AND the commit message or branch indicates a verification plan, require `required_verification` skills in addition to `required_planning`
3. The check is gated on WORKFLOW.md: only fire when the current FLOW in WORKFLOW.md is a verification flow

**Implementation complexity**: Medium. Requires:
- New config key in templates/silver-bullet.config.json.default
- completion-audit.sh Tier 1 path extended (~20 lines)
- WORKFLOW.md current-flow detection (already partially built in prompt-reminder.sh)

**Why implementation is deferred**: The task boundary signal (commit message / FLOW detection) requires reliable WORKFLOW.md parsing. The current workflow-utils.sh parses flow counts but not the active flow name. Implementing that parser is a separate task. This design doc specifies the target state; the parser work is tracked separately.

### References
- hooks/completion-audit.sh — Tier 1/Tier 2 logic
- hooks/stop-check.sh — final delivery gate
- hooks/lib/workflow-utils.sh — flow log parsing utilities
- GitHub issue #72

## Title: FLOW-01 — FLOW Layer Parallelism in /silver Composer: Design Note

### Current State
The /silver composer (skills/silver/SKILL.md) routes user intent to a single workflow skill and invokes it sequentially. Each silver-* workflow (silver-feature, silver-ui, silver-bugfix, etc.) is itself a sequential FLOW composition: DISCUSS → PLAN → EXECUTE → VERIFY → REVIEW → FINALIZE → SHIP. There is no parallelism at the FLOW layer today.

### What FLOW Layer Parallelism Means
A "FLOW" in Silver Bullet terminology is one composable path through the workflow (e.g., FLOW 1 = DISCUSS, FLOW 2 = PLAN, ...). FLOW layer parallelism means running two or more FLOWs simultaneously, or running sub-FLOWs within a single workflow path in parallel.

Two forms of parallelism are relevant to the /silver composer:

**Form 1: Multi-feature parallelism** — Two independent silver-* workflows running concurrently (e.g., silver-feature for Feature A and silver-bugfix for Bug B). Signals that would trigger this:
- User input contains "and also" / "while also" / "in parallel with"
- User explicitly names two independent features or bugs
- MultAI is available (each independent flow runs in a separate AI process)

**Form 2: Intra-workflow FLOW parallelism** — Within a single silver-* workflow, certain FLOWs can run concurrently. For example, after EXECUTE completes, VERIFY (automated tests) and REVIEW (code quality) could run in parallel since they are independent of each other.

### Dependency Model
For FLOW layer parallelism to be safe, a dependency model is required:

| FLOW | Depends on | Can parallel with |
|------|-----------|-------------------|
| DISCUSS | — | Nothing (user decision required) |
| PLAN | DISCUSS | Nothing (sequential by design) |
| EXECUTE | PLAN | Nothing (writes to codebase, exclusive) |
| VERIFY | EXECUTE | REVIEW (both read-only after EXECUTE) |
| REVIEW | EXECUTE | VERIFY (both read-only after EXECUTE) |
| FINALIZE | VERIFY + REVIEW | Nothing (gathers results) |
| SHIP | FINALIZE | Nothing (external action) |

The only safe parallelism in the current workflow model is VERIFY || REVIEW after EXECUTE completes.

### Implementation Prerequisites (all deferred)
1. MultAI integration: ability to spawn two AI agents and collect both results
2. WORKFLOW.md parser: must detect which FLOWs are complete vs in-progress
3. Merge protocol: when two parallel FLOWs both write findings (e.g., VERIFY and REVIEW both produce reports), the FINALIZE flow must merge them
4. Conflict detection: two FLOWs must not write the same file simultaneously

### Signal Design for the Composer
When /silver receives input, it should detect parallelism signals before routing:
- Parse input for "and also" / "simultaneously" / "in parallel" → propose multi-flow execution
- After EXECUTE completes in any workflow, automatically offer VERIFY+REVIEW in parallel if MultAI is available
- Display: "VERIFY and REVIEW can run in parallel. Run them together? (A. Yes / B. Sequential)"

### References

- skills/silver-feature/SKILL.md — FLOW composition template
- GitHub issue #75
