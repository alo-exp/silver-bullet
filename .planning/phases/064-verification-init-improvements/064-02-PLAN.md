## Title: BUG-06 — Claude Code Permission Re-Prompting After Bypass Permissions: Investigation

### Bug Description
After the user sets "Bypass Permissions" in Claude Code, Claude should not re-prompt for permissions when running shell commands, file reads, etc. However, the bug report (GitHub #64) indicates that Claude Code re-prompts for permissions even after Bypass Permissions is set. The question is whether any Silver Bullet hook is triggering this behavior.

### Hooks Investigated
Document each hook checked:

For each hook, state:
- Hook event type
- Output format used (additionalContext / permissionDecision:deny / decision:block)
- Whether it could trigger a permission dialog

**Key finding to document:**

The `permissionDecision:"deny"` output format in Claude Code hooks is the mechanism that triggers permission dialogs. Hooks that emit:
```json
{"hookSpecificOutput":{"hookEventName":"PreToolUse","permissionDecision":"deny","permissionDecisionReason":"..."}}
```
cause Claude Code to show a "permission denied" dialog. Silver Bullet uses this format in two places:

1. **forbidden-skill-check.sh** (PreToolUse/Skill): emits `permissionDecision:"deny"` when a forbidden skill is invoked. This is intentional — it should block the skill. However, if the Claude Code platform interprets this as a "permission denied" event that persists after Bypass Permissions is set, it could be the source of the re-prompt bug.

2. **completion-audit.sh** (PreToolUse/Bash): its `emit_block()` function emits `permissionDecision:"deny"` for PreToolUse events. If the commit being blocked is a git command, and Bypass Permissions is set, the "deny" response from completion-audit.sh might interact with the Bypass Permissions state.

**Stop hook** (stop-check.sh) uses `"decision":"block"` which is the Stop event format — not a permission dialog trigger.

**prompt-reminder.sh** (UserPromptSubmit) uses `additionalContext` only — no permission relevance.

**session-start** uses `additionalContext` only — no permission relevance.

### Root Cause Assessment
Document the assessment with one of these verdicts:

**If the verdict is "SB hooks may contribute":**
The `permissionDecision:"deny"` output from `forbidden-skill-check.sh` and `completion-audit.sh` is the technically correct format per the Claude Code hook protocol for PreToolUse hooks that need to block tool use. Whether these "deny" decisions interact with the Bypass Permissions setting is a platform behavior question. SB cannot change the protocol it uses to block tools without losing the ability to enforce the forbidden-skill gate.

Potential mitigation (if within SB control): Check whether there is an alternative output format for PreToolUse blocking that does NOT trigger a permission re-prompt. If the Claude Code hook docs specify a non-dialog blocking format, update the hooks to use it.

**If the verdict is "platform issue":**
The permission re-prompting behavior after Bypass Permissions is set is a Claude Code platform issue. Silver Bullet's `permissionDecision:"deny"` responses are the correct hook output format; the platform should honor Bypass Permissions and not re-prompt. Silver Bullet cannot fix this without a platform-level change.

### Disposition
State one of:
- "Fix applied: [description of code change]" (if a hook change was made)
- "Platform issue: No SB fix possible. GitHub issue #64 should be updated with these findings. Link: https://github.com/alo-exp/silver-bullet/issues/64"
