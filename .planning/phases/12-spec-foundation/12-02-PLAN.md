**Frontmatter** (YAML between triple-dash delimiters, same format as silver-feature):
  name: silver-spec
  description: "AI-guided Socratic spec elicitation: interactive dialogue producing SPEC.md + REQUIREMENTS.md from scratch or augmenting an existing draft"
  argument-hint: "<feature name or description>"

**Title and intro:**
  # /silver:spec -- Spec Elicitation Workflow
  SB orchestrator for requirements elicitation. Guides PM/BA through structured Socratic dialogue to produce canonical .planning/SPEC.md and .planning/REQUIREMENTS.md artifacts.
  Never implements features directly -- orchestrates dialogue and writes spec artifacts only.

**Pre-flight: Load Preferences** -- same pattern as silver-feature: read silver-bullet.md section 10, silently apply preferences. Display banner using the box art format from silver-feature, with title "SILVER BULLET > SPEC ELICITATION", showing Feature ($ARGUMENTS or "(not specified)") and Mode (greenfield | augment).

**Step-Skip Protocol** -- same pattern as silver-feature. Non-skippable gates: `Step 3: Socratic Elicitation`, `Step 5: Assumption Consolidation`, `Step 7: Write SPEC.md`.

**Step 0: Mode Detection**
- If `.planning/SPEC.md` exists: augment mode (refine existing spec). Read existing spec, show current spec-version and section summary.
- Else: greenfield mode (create from scratch).

**Step 1: Context Gathering**
Ask for (one prompt, lettered options for optional items):
- Feature name (required)
- Feature description (required -- 1-2 sentences)
- A. JIRA ticket ID (optional)
- B. Figma URL (optional)
- C. Google Doc / PPT URL (optional)

If any URL is provided, note it for artifact injection in Step 4.

**Step 2: Invoke product-management:write-spec (ELIC-06)**
Invoke `product-management:write-spec` via the Skill tool. This generates a formal PM spec scaffold that the Socratic dialogue will fill. If the skill is unavailable (invocation fails), proceed without it -- the template provides structure.

**Step 3: Socratic Elicitation Dialogue**
Run 9 questioning turns in sequence. After EACH answer from the PM/BA:
1. State any implicit assumption: "I'm assuming [X] -- is that right?"
2. If PM/BA says "I don't know yet" or cannot resolve: emit `[ASSUMPTION: {what SB is assuming} | Status: Follow-up-required | Owner: TBD]`
3. Ask: "Anything else on this topic, or move to the next?"

Turn sequence (include the exact question patterns in the SKILL.md):

| Turn | Domain | Question |
|------|--------|----------|
| 1 | Problem | "What problem does this solve? For whom?" |
| 2 | User goal | "When a user reaches this feature, what do they want to accomplish?" |
| 3 | Scope boundary | "What is explicitly OUT of scope for this feature?" |
| 4 | User stories | "Walk me through the main thing a user does with this feature, step by step" |
| 5 | Acceptance criteria | "How do we know this works correctly? List one criterion at a time" |
| 6 | Edge cases | "What happens when [common failure scenario]?" |
| 7 | Error states | "What should the user see when something goes wrong?" |
| 8 | Data model | "What data does this feature create, read, update, or delete?" |
| 9 | Open questions | "What do you not know yet that would affect the spec?" |

**Step 4: Artifact Injection (conditional -- only if URL provided in Step 1)**
- Display URL, attempt text extraction (WebFetch for Google Docs, Skill tool `design:user-research` for Figma)
- Show 3-bullet extraction summary
- Ask: "A. Incorporate into spec  B. Skip"
- If Figma URL: invoke `design:user-research` via Skill tool (ELIC-06)
- If Google Doc/PPT URL: attempt text extraction via WebFetch tool. If inaccessible, record URL in source-artifacts frontmatter for Phase 13 MCP ingestion.

**Step 5: Assumption Consolidation**
- Collect all `[ASSUMPTION: ...]` blocks surfaced during Steps 3-4
- Present numbered list to PM/BA
- For each: ask "A. Resolve now  B. Accept as assumption  C. Tag for follow-up"
- Update status field accordingly (Resolved / Follow-up-required)

**Step 6: Invoke design:design-critique (conditional -- only if design artifact exists) (ELIC-06)**
Invoke `design:design-critique` via Skill tool. If unavailable, skip with note.

**Step 7: Write .planning/SPEC.md**
- Read `templates/specs/SPEC.md.template`
- Populate all sections from elicitation answers collected in Steps 1-6
- Set frontmatter:
  - `spec-version`: 1 for greenfield, existing value +1 for augment
  - `status`: Draft
  - `jira-id`: from Step 1 if provided, else empty string
  - `figma-url`: from Step 1 if provided, else empty string
  - `source-artifacts`: list of URLs provided in Step 1
  - `created`: today's date (greenfield) or preserve existing (augment)
  - `last-updated`: today's date
- Write using the Write tool to `.planning/SPEC.md`
- In augment mode: read existing `spec-version` first, increment by 1

**Step 8: Write .planning/REQUIREMENTS.md**
- Read `templates/specs/REQUIREMENTS.md.template`
- Derive REQ-XX IDs from acceptance criteria in Step 5
- Derive NFR-XX IDs from any non-functional concerns raised during elicitation
- Write using the Write tool to `.planning/REQUIREMENTS.md`

**Step 9: Write .planning/DESIGN.md (conditional -- only if design artifact or Figma URL was provided)**
- Read `templates/specs/DESIGN.md.template`
- Populate from design context gathered in Steps 4 and 6
- Write using the Write tool to `.planning/DESIGN.md`

**Step 10: Commit artifacts**
Run git add for .planning/SPEC.md, .planning/REQUIREMENTS.md, .planning/DESIGN.md (with 2>/dev/null for DESIGN.md in case it was not created), then git commit with message "spec: [feature-slug] v{spec-version} draft"

**Step 11: Summary**
Display summary banner (same box art style) showing: spec-version, section count, assumption count, open question count, status (Draft), and next step ("run /silver:feature to begin implementation planning").
