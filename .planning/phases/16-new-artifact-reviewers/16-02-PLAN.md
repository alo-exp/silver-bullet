**skills/review-context/SKILL.md** (ARVW-05):
- Frontmatter: name: review-context, description of CONTEXT.md reviewer, argument-hint: "&lt;context-path&gt;"
- Loading Rules: MUST load @skills/artifact-reviewer/rules/reviewer-interface.md and @skills/artifact-reviewer/rules/review-loop.md
- Quality Criteria:
  1. Decisions section exists and is non-empty
  2. Every gray area / discussion point has a resolution — either a locked decision (with specific choice, not vague like "we'll figure it out") or explicitly marked as "Claude's Discretion"
  3. Decision specificity: each locked decision contains a concrete choice (e.g., "use PostgreSQL" not "use a database") — vague decisions produce ISSUE findings
  4. No contradictions: no two decisions in the same CONTEXT.md contradict each other (e.g., "use REST API" and "use GraphQL" for the same endpoint)
  5. Deferred Ideas section: if present, items are clearly separated from active decisions (no deferred idea appearing in Decisions)
  6. Claude's Discretion items have sufficient context for Claude to make a reasonable choice
- Finding ID prefix: "CTX-F"
- Output Contract and Prohibitions: per reviewer-interface.md

**skills/review-research/SKILL.md** (ARVW-06):
- Frontmatter: name: review-research, description of RESEARCH.md reviewer, argument-hint: "&lt;research-path&gt;"
- Loading Rules: same framework rules
- Quality Criteria:
  1. Research addresses the phase's key questions (if phase context available via review_context)
  2. Findings are evidence-based: each finding cites a source (documentation URL, library version, benchmark data) — findings without evidence produce ISSUE findings with description "Speculative finding — no evidence cited"
  3. Confidence levels (if present) are justified with reasoning, not just asserted
  4. Pitfalls / "don't hand-roll" warnings are actionable — they name the specific risk and the alternative
  5. Recommendations section exists with clear, implementable choices
  6. No stale references: version numbers mentioned should be current (INFO-level if potentially outdated)
- Finding ID prefix: "RES-F"
- Output Contract and Prohibitions: per reviewer-interface.md

**skills/review-ingestion-manifest/SKILL.md** (ARVW-07):
- Frontmatter: name: review-ingestion-manifest, description of INGESTION_MANIFEST.md reviewer, argument-hint: "&lt;manifest-path&gt; [--source-inputs &lt;spec-path&gt;]"
- Loading Rules: load reviewer-interface.md and review-loop.md
- Quality Criteria:
  1. All source artifacts listed: every artifact referenced in the ingestion run has a row in the manifest
  2. Status accuracy: each row has status (succeeded / failed / missing) — no blank statuses
  3. Failed artifact correspondence: every artifact with status "failed" or "missing" has a corresponding `[ARTIFACT MISSING: reason]` block in the linked SPEC.md (when source_inputs includes spec-path). If a failed artifact has no corresponding MISSING block, produce ISSUE finding
  4. No false success: if an artifact status is "succeeded" but the ingested content in SPEC.md is empty or placeholder, produce ISSUE finding
  5. Reason field: failed/missing artifacts have a non-empty reason explaining the failure
  6. Resumability: manifest includes enough state (artifact IDs, timestamps) for silver-ingest to resume from last success
- Finding ID prefix: "INGM-F"

**skills/review-uat/SKILL.md** (ARVW-08):
- Frontmatter: name: review-uat, description of UAT.md reviewer, argument-hint: "&lt;uat-path&gt; [--source-inputs &lt;spec-path&gt;]"
- Loading Rules: load reviewer-interface.md and review-loop.md
- Quality Criteria:
  1. Coverage: every Acceptance Criterion from the linked SPEC.md has a corresponding UAT row — missing criteria produce ISSUE findings
  2. No orphaned UAT rows: every UAT row traces to a SPEC.md Acceptance Criterion
  3. Evidence quality: pass/fail evidence is substantive — "looks good", "works", "tested" without detail produce ISSUE findings. Evidence must describe what was observed or measured.
  4. Spec-version match: UAT.md's spec-version field matches the linked SPEC.md's spec-version — mismatches produce ISSUE findings
  5. No blank results: every UAT row has a pass/fail status and evidence field filled
  6. Fail follow-up: any row marked "fail" should have a linked issue or remediation note
- Finding ID prefix: "UAT-F"
