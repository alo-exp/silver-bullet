**Test 8 — atomicity under 10 parallel claims (LOCK-05 case 7):**
- `reset_lock`
- Launch 10 background `"$SCRIPT" claim 070 claude "race-$i"` jobs in parallel:
  ```bash
  successes=0
  conflicts=0
  pids=()
  rcs_dir=$(mktemp -d)
  for i in $(seq 1 10); do
    ( "$SCRIPT" claim 070 claude "race-$i" >/dev/null 2>&1; echo $? > "$rcs_dir/$i" ) &
    pids+=($!)
  done
  for p in "${pids[@]}"; do wait "$p" || true; done
  for i in $(seq 1 10); do
    rc=$(cat "$rcs_dir/$i")
    if [[ "$rc" == "0" ]]; then successes=$((successes+1))
    elif [[ "$rc" == "2" ]]; then conflicts=$((conflicts+1))
    fi
  done
  rm -rf "$rcs_dir"
  ```
- Assert `successes == 1` (PASS) else FAIL with diagnostic showing actual counts.
- Assert `conflicts == 9` (PASS).
- Note on same-runtime races: all 10 claim attempts use `runtime=claude`, but since each subprocess has a distinct PID, the owner-ids differ and one becomes the owner — the other 9 see a foreign owner-id and exit 2 (conflict). This is the correct atomicity probe. (If the helper treated same-runtime claims as idempotent, this assertion would fail — that's the wrong behavior; the test catches it.)
