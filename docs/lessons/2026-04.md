# Lessons Learned — 2026-04

## stack:bash

2026-04-05 — Shell hooks that parse JSON config on every invocation add measurable latency. Prefer hardcoded values for hot-path hooks and accept the sync burden, unless invocation frequency is low enough that runtime config parsing is acceptable.

## practice:enforcement

2026-04-05 — When promoting a workflow step from "recommended" to "enforced," the change is breaking for all existing users — their workflows will be blocked until they comply. Treat enforcement promotions as semver minor bumps and document prominently.

2026-04-05 — Enforcement systems that use separate lists for "tracked" vs "required" give a useful graduation path: observe first (track), then enforce (require). This two-tier model reduces false positives during rollout.

## practice:config-management

2026-04-05 — When a config file has a template copy (for new project scaffolding), both files must stay in exact parity. Without CI enforcement, this drift is discovered only during code review — add a `diff` assertion in CI to catch it automatically.

2026-04-05 — Conditional enforcement rules (e.g., "only required when UI work is present") must NOT be added to unconditional gate lists. The gate has no way to evaluate the condition — it just blocks everything.


2026-04-14 — Whitelisted single-marker state-file appends are rejected when chained with `&&`. The whitelist regex matches only `^echo '<marker>' >> <state-file>$` and a separate check immediately revokes the whitelist if the command contains any chaining operator (`&&`, `||`, `;`, `|`). Write each marker as a separate Bash invocation — there is no shortcut.

2026-04-24 — Shell tools that emit ANSI color codes silently break text processing. Commands like `forge config list` use `\x1b\[[0-9;]*m` escape sequences for colored output. When you `grep "^max_tokens = "` on this output, it matches nothing — the literal text in the output is `"<ESC>[1;32mmax_tokens = 196608<ESC>[m"`, not `"max_tokens = 196608"`. Always strip ANSI codes first: pipe through `sed 's/\x1b\[[0-9;]*m//g'` or use `--color=never` flags before parsing. This bites in CI scripts and config validation where the failure is silent.

## stack:forge

2026-04-24 — Forge's model configuration lives in a TOML file (`~/.claude/plugins/cache/forge*/config/model.toml` or similar). The settings include `model`, `max_tokens`, and provider endpoint. After editing this file, forge picks up the changes on next invocation — no restart needed.

2026-04-24 — To verify forge's active configuration, use `forge config list` and `forge config get <key>`. When parsing `config list` programmatically, strip ANSI codes first:
```bash
forge config list 2>/dev/null | sed 's/\x1b\[[0-9;]*m//g' | awk 'NR==27 {print $3}'
```
The line number (NR==27) is fragile but works when grep fails due to color codes.

2026-04-24 — Forge uses MiniMax-M2.7 via `https://api.minimax.io/anthropic/v1/messages`. The `max_tokens` setting controls response length. Increasing from a lower value to 196608 (192K tokens) allows much longer responses — useful for stress-testing or generating large outputs.

2026-04-24 — When testing API changes via forge, run a small prompt first to confirm basic connectivity, then a larger one to verify the new settings take effect. Timeouts on large responses may indicate the new `max_tokens` is working (the API is trying to return more data).

2026-04-24 — Create a dedicated test script in `tests/forge-test-app/` for repeatable verification:
```bash
#!/bin/bash
MODEL=$(forge config list 2>/dev/null | sed 's/\x1b\[[0-9;]*m//g' | awk 'NR==25 {print $3}')
MAX_TOKENS=$(forge config list 2>/dev/null | sed 's/\x1b\[[0-9;]*m//g' | awk 'NR==27 {print $3}')
echo "Model: $MODEL, max_tokens: $MAX_TOKENS"
[[ "$MAX_TOKENS" == "196608" ]] && echo "PASS" || echo "FAIL"
```
Keep test scripts executable (`chmod +x`) and run them as part of CI.
