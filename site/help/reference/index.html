GSD Commands

GSD commands are slash commands that resolve to GSD's installed workflow files. All are invoked via the Skill tool inside Claude Code.

CommandPhaseDescription
/gsd:new-project Init Kick off a new project: ecosystem research, requirements scoping, roadmap generation. Produces .planning/PROJECT.md, REQUIREMENTS.md, ROADMAP.md.
/gsd:discuss-phase Discuss Capture implementation decisions and gray areas for the current phase before planning. Produces .planning/{phase}-CONTEXT.md.
/gsd:plan-phase Plan Research → task plan → plan verification. Produces .planning/{phase}-RESEARCH.md and .planning/{phase}-{N}-PLAN.md.
/gsd:execute-phase Execute Wave-based parallel execution with atomic commits per task. Produces .planning/{phase}-{N}-SUMMARY.md and git commits.
/gsd:verify-work Verify Goal-backward verification against requirements + UAT. Produces .planning/{phase}-VERIFICATION.md.
/gsd-ship Ship Create PR with auto-generated body: phase summaries, requirement coverage, verification links.
/gsd:next Any "Where am I?" — reads current state and advances to the next required step. Use when lost.
/gsd:debug Any Systematic debugging for any bug encountered during execution. Uses scientific method: hypothesize → test → confirm.
/gsd:help Any Displays the complete GSD command reference.

Silver Bullet Skills

Skills required by Silver Bullet's workflows. /silver is the smart router — the single entry point that dispatches to any SB or GSD skill. Quality gates, blast radius, forensics, release notes, and /silver:init are Silver Bullet's own. Deploy checklist, testing strategy, tech debt, documentation, and finishing-a-development-branch come from Superpowers and Engineering plugins that Silver Bullet enforces.

SkillWhenDescription
/silver Anytime Smart router — accepts freeform natural language and dispatches to the most appropriate Silver Bullet or GSD skill. Performs complexity triage (Trivial / Simple / Complex / Fuzzy) before routing. See Complexity Triage below. Use when you're unsure which skill to invoke.
/quality-gatesRequired Before Plan 9-dimension quality evaluation (modularity, reusability, scalability, security, reliability, usability, testability, extensibility, plus devops-quality-gates for infra-touching changes). Dispatches parallel agents. All must pass.
/blast-radiusDevOpsRequired Before Plan (DevOps) Maps change scope, dependencies, failure scenarios, rollback plan, and change window risk. Produces LOW/MEDIUM/HIGH/CRITICAL rating.
/devops-quality-gatesDevOpsRequired Before Plan (DevOps) IaC-adapted quality gates — 7 dimensions (usability excluded). Same base framework as /quality-gates, adapted for infrastructure: idempotency, least-privilege IAM, drift prevention. Dispatches 7 parallel agents.
/accessibility-reviewUI onlyRequired Discuss (UI phases) WCAG 2.1 AA compliance review. Required for any phase that introduces or modifies user-facing UI. Checks colour contrast, keyboard nav, screen reader compatibility, focus management.
/deploy-checklistRequired Before ship Pre-deployment verification gate. Checks rollback plan, monitoring, stakeholder sign-off, CI status.
/create-releaseRequired After ship Generates git tag and GitHub Release. Checks README is current before proceeding.
/testing-strategyRequired Finalization Define test pyramid, coverage goals, tooling. Produces testing strategy document.
/tech-debtRequired Finalization Identify, categorize, and prioritize technical debt introduced or surfaced during the work. Appends structured items to docs/tech-debt.md: | Item | Severity | Effort | Phase introduced |.
/documentationRequired Finalization Update or create all project docs: README, PRD-Overview, Architecture, Testing, CICD, KNOWLEDGE, CHANGELOG.
/finishing-a-development-branchRequired Finalization Branch rebase, cleanup, and merge preparation. Skipped automatically when working directly on main/master.
/incident-responseDevOpsRequired Incident Fast Path (Step 1) Establish severity, timeline, communication plan, and rollback criteria before touching anything. Required first step for any active production incident.
/forensics On failure Root-cause investigation for completed, failed, or abandoned sessions. Use after /gsd:verify-work fails.
/silver:init Setup Initialize or re-activate Silver Bullet for a project. Idempotent on re-run.
/silver:feature Feature work Orchestrated feature development workflow. Chains intel → explore (if fuzzy) → product-brainstorm → silver:brainstorm → quality-gates → GSD plan/execute → TDD → code review → verify → ship. Routed automatically by /silver on "add", "build", "implement", "new feature" signals.
/silver:bugfix Bug fixes Triage-first bug investigation. Three paths: 1A (known symptom, unknown fix), 1B (silver:forensics for reconstruction), 1C (GSD-specific diagnostics). Includes mandatory TDD regression test before implementing fix. Routed on "bug", "broken", "crash", "error", "regression" signals.
/silver:ui UI/UX work UI/UX workflow with mandatory gsd-ui-review gate (6-pillar visual audit including WCAG accessibility). Chains intel → product-brainstorm → silver:brainstorm → quality-gates → gsd-ui-phase → gsd-ui-review → code review → verify → ship. Routed on "UI", "frontend", "component", "screen", "design", "layout" signals.
/silver:devops Infrastructure DevOps/IaC workflow with mandatory blast-radius assessment. Uses 7 IaC-adapted quality dimensions. Chains intel → blast-radius → devops-skill-router → devops-quality-gates → GSD plan/execute → code review → verify → deploy checklist → ship. Routed on "infra", "CI/CD", "deploy", "terraform", "kubernetes" signals.
/silver:research Research/spike Technology decisions and architecture spikes. Chains silver:explore → MultAI research (proactive offer) → silver:brainstorm → decision with rationale → hand off to implementing workflow. Routed on "how should we", "compare X vs Y", "spike", "which technology" signals.
/silver:release Release prep Milestone-level publish workflow (NOT phase-level ship). Chains quality-gates → silver:security → gsd-audit-uat → gsd-audit-milestone → deploy-checklist → silver:create-release (git tag + GitHub Release) → gsd-ship → gsd-complete-milestone. Routed on version number, "release", "publish", "go live" signals.
/silver:fast Trivial changes Bypass workflow for trivial changes (≤3 files, typos, config tweaks, renames). Routes directly to gsd-fast — no planning overhead. §10 routing preferences NOT applied by design. Also invoked automatically by /silver when complexity triage classifies as Trivial.
/silver:update Session start (auto) or anytime Fetches the latest Silver Bullet version and requires commit SHA confirmation before applying. Invoked automatically at session start when the installed version is behind latest. Manual invocation: /silver:update.

Spec Pipeline Skills (v0.14.0+)

Three skills that form the spec-driven development pipeline. They run before GSD planning begins — SPEC.md is the upstream input that REQUIREMENTS.md, ROADMAP.md, and DESIGN.md must stay aligned with.

SkillWhenDescription
/silver:spec Before planning (spec-first projects) AI-driven spec elicitation. Produces a canonical SPEC.md with structured requirements, acceptance criteria, assumption tracking, and source input cross-references. Use before /gsd:new-project or when a formal spec is required before GSD planning.
/silver:ingest Before spec creation (external sources) Ingests artifacts from JIRA, Figma, and Google Docs via MCP connectors. No custom API code — all external data access delegated to MCP. Produces a structured INGESTION_MANIFEST.md that maps external inputs to spec sections.
/silver:validate After spec creation, before planning Runs three validation gates: spec floor check (spec-floor-check.sh) for SPEC.md completeness, PR-to-spec traceability (pr-traceability.sh), and UAT pipeline gate (uat-gate.sh). All three must pass before planning can proceed.
/silver-review-stats Anytime (review health reporting) Generates three summary tables from accumulated review metrics (JSON Lines): pass rates by artifact type, rounds to clean pass, and finding categories by artifact type. Use to track review health over time.

Orchestration Workflows

Eight pre-designed workflows invoked automatically by /silver after intent classification. Each workflow chains GSD, Superpowers, MultAI, and Silver Bullet quality gates in a defined sequence. They orchestrate — they never implement directly.

SkillEntry trigger keywordsFirst step
/silver:feature add, build, implement, new feature, enhance, extend Complexity triage → silver:intel (brownfield) → silver:explore (fuzzy) → product-brainstorm
/silver:bugfix bug, broken, crash, error, regression, failing test, not working Triage path selection (A/B/C) → systematic-debugging or silver:forensics
/silver:ui UI, frontend, component, screen, design, interface, layout, animation, responsive Complexity triage → silver:intel → silver:explore (if fuzzy) → product-brainstorm
/silver:devops infra, CI/CD, deploy, pipeline, terraform, IaC, kubernetes, container, cloud, ops Complexity triage → silver:intel → blast-radius assessment
/silver:research how should we, which technology, compare X vs Y, spike, investigate, architecture decision Classify (research vs. feature/bugfix?) → silver:explore → MultAI research offer
/silver:release release, publish, version, go live, cut a release, tag v, ship to users, deploy to prod Ship disambiguation check → silver:quality-gates (pre-release full pass)
/silver:fast trivial, quick fix, typo, one-liner, config value, rename, ≤3 files Confirm trivial (≤3 files?) → gsd-fast execution → atomic commit

Complexity Triage

/silver classifies every request before routing. This prevents over-engineering trivial changes and under-engineering complex ones.

ClassificationSignalsAction
TrivialTypo, config, rename, ≤3 filesRoute to /silver:fast (gsd-fast) — bypass workflow
SimpleClear scope, ≤1 phaseRoute to workflow, skip silver:explore
ComplexMulti-phase, cross-cuttingFull workflow including silver:explore + brainstorm
FuzzyVague intent, unclear scopeRoute to /silver:explore first, then re-classify

Ship Disambiguation

/silver distinguishes between phase-level ship (gsd-ship) and milestone-level release (silver:release) based on context signals.

SignalRoute
Contains version number (v2.0, 1.4.0…)/silver:release
Contains "changelog" or "release notes"/silver:release
Contains "go live", "to production", "publicly"/silver:release
Active phase in progress, no version signal/gsd-ship (phase merge)
No active phase, end of milestone/silver:release

Superpowers Skills Used in Silver Bullet

SkillWhenDescription
/brainstormingSP Before Discuss Explores problem space. After completion, redirect handoff to /gsd:plan-phase — not to writing-plans.
/system-designSPCond Discuss (new service) Architecture decisions for new services or major components. Required when a phase introduces significant new infrastructure.
/design-systemSPCond Discuss (UI work) UI component and design token decisions. Only required for phases with UI work.
/ux-copySPCond Discuss (UI work) User-facing text and messaging decisions. Only required for phases with UI work.
/test-driven-developmentSPRequired Execute (before writing code) Red-green-refactor discipline. Invoke before writing any implementation code in /gsd:execute-phase. Write a failing test first, then implement until it passes, then refactor. Required in both dev and DevOps (Terratest / conftest / OPA / BATS for IaC).
/code-reviewENGRequired Code Review Structured peer code quality review: security (SQL injection, XSS, CSRF, auth flaws), performance (N+1, unbounded queries), correctness (edge cases, race conditions), maintainability. Run before /requesting-code-review.
/requesting-code-reviewSPRequired Code Review Dispatches superpowers:code-reviewer for peer code quality review: security, performance, correctness, readability. Loop until 2 consecutive Approved passes.
/receiving-code-reviewSPRequired Code Review Triage all review feedback. Accept or reject each item with reasoning before fixing.
superpowers:code-reviewer Code Review (via /requesting-code-review) Automated reviewer subagent dispatched by /requesting-code-review. Must return Approved on two consecutive passes before proceeding. Self-limiting — loop ends naturally when two clean passes are produced.

.silver-bullet.json Configuration

Project-level configuration file in the repo root. Created by /silver:init.

{ "project": { "name": "my-app", "src_pattern": "/src/", // source root for semantic compression "src_exclude_pattern": "__tests__|\\.test\\." // files to skip }, "semantic_compression": { "enabled": true, // set false to disable "context_budget_kb": 50, // max KB of context to inject "min_file_size_bytes": 3072, // files below this are included in full "top_chunks_per_file": 3, // max chunks to include per large file "chunk_size_bytes": 1024, // chunk size for TF-IDF scoring "debug": false // set true to log to .planning/.context-cache/debug.log }, "verify_commands": [ // local verify commands run at CI gate (step 17) "npm test", "npm run lint" ], "all_tracked": [ // skills always cross-referenced at skill discovery "security", "performance" ], "workflows": { "default": "docs/workflows/full-dev-cycle.md", "devops": "docs/workflows/devops-cycle.md" } }

Session Log Format

A session log is created at the start of every session at docs/sessions/YYYY-MM-DD-{task-slug}.md. The path is written to ~/.claude/.silver-bullet/session-log-path for hooks to access.

# Session: {task-slug} ## Meta Date: YYYY-MM-DD Model: claude-sonnet-4-6 Mode: interactive | autonomous ## Pre-answers - Model routing — Planning: Sonnet - Worktree: yes ## Task {one-sentence description} ## Approach {how you approached it} ## Skills flagged at discovery {skills surfaced during skill discovery step} ## Skill gap check (post-plan) {any skills flagged after plan was written} ## Files changed {list of files modified} ## Skills invoked {ordered list of skills actually invoked} ## Agent Teams dispatched {agents dispatched, isolation mode, results} ## Autonomous decisions {decisions made without user input in autonomous mode} ## Outcome {what was achieved} ## KNOWLEDGE.md additions updated (Architecture patterns, Key decisions) | no changes ## Virtual cost ~$0.45 (Sonnet, medium complexity)

State Files (~/.claude/.silver-bullet/)

These files live in ~/.claude/.silver-bullet/ with restricted permissions (0700). They store session state for hooks.

FilePurpose

~/.claude/.silver-bullet/session-log-pathPath to the current session log file in docs/sessions/. Used by hooks to write events.
~/.claude/.silver-bullet/trivialPresence signals that the next commit is a trivial change. Bypass workflow enforcement for one commit.
~/.claude/.silver-bullet/session-initSentinel file marking that the session init phase (loading context, compacting) has already run.

Useful Shortcuts and Patterns

Common Command Sequences

# Start a new feature on an existing project /gsd:discuss-phase → /quality-gates → /gsd:plan-phase → /gsd:execute-phase # Continue where you left off /gsd:next # Something broke during execution /gsd:debug # Verify and ship after execution /gsd:verify-work → /code-review → /requesting-code-review → /receiving-code-review # Mark next commit as trivial (no enforcement) touch ~/.claude/.silver-bullet/trivial # Check CI status /opt/homebrew/bin/gh run list --limit 1 --json status,conclusion,name,headBranch # Full finalization + release sequence /testing-strategy → /documentation → /finishing-a-development-branch → /deploy-checklist → /gsd-ship → /create-release

Model Routing Guidance

Silver Bullet uses claude-sonnet-4-6 by default. You're asked about Opus at two transition points only:

  • Before the Planning phase begins (Discuss step) — for deeper reasoning on complex architectures
  • Before Design sub-steps (design-system, ux-copy, architecture, system-design) — for richer design output

In Autonomous mode, Sonnet is used throughout. Opus is escalated silently only if planning output is measurably incomplete (fewer than 5 lines, contains TBD or [TODO] placeholders, or a file-producing step produces no file).
