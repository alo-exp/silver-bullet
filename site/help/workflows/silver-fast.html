/silver:fast

Trivial change fast path — bypass workflow for ≤3 file changes: typos, config values, renames. Routes directly to gsd-fast with no planning overhead.

Overview

/silver:fast is the Silver Bullet bypass workflow for trivial changes. It skips all workflow overhead — no planning, no quality gates, no brainstorming, no review cycle — and routes directly to gsd-fast execution.

Appropriate for: typo fixes, config value changes, symbol renames ≤3 files, one-liner code changes, comment updates.

Also invoked automatically by /silver when complexity triage classifies the request as Trivial (≤3 files, clear trivial scope).

ℹ
§10 preferences not applied: This workflow deliberately does NOT read §10 user preferences. The fast path skips all preference overhead by design — preference loading adds latency that defeats the purpose of a trivial bypass.

When to use

Entry trigger signals for /silver:fast:

  • "trivial" / "quick fix" / "typo" / "one-liner" / "config value"
  • ≤3 files, rename, comment update
  • Automatically invoked by /silver when complexity triage classifies as Trivial

Scope limit

⚠
silver:fast is strictly for ≤3 file changes. If the work expands beyond this limit during execution, the fast path STOPS immediately and offers escalation to a named workflow. False trivial classifications cause under-reviewed changes. When scope is ambiguous: escalate.

Workflow steps

Pre-flight: banner only

Displays a fast-path banner. No preference loading (intentional).

Step 0 — Complexity triage gate

Before invoking gsd-fast, SB confirms the change is truly trivial:

Confirming trivial scope for: {change description} Does this change: A. Touch ≤3 files AND is a typo / config value / rename / one-liner? → proceed to fast path B. Touch >3 files OR involves logic, dependency, or schema changes? → escalate to full workflow

If A (trivial confirmed): proceed to Step 1.

If B (non-trivial): do NOT invoke gsd-fast. Instead, SB asks which workflow to route to (silver:feature / silver:bugfix / silver:ui / silver:devops / silver:research) and invokes the chosen workflow with the original arguments.

Step 1 — Execute fast path

Only reached if Step 0 confirms trivial scope. Invoke gsd-fast with the change description as argument.

Step 2 — Scope expansion stop condition

During or after gsd-fast execution, if scope expands beyond 3 files, the fast path stops immediately:

Scope has expanded beyond 3 files. Fast path cannot continue. Files now affected: {list}

SB then asks which workflow should take over (silver:feature / silver:bugfix / silver:devops / stop for manual review). The chosen workflow receives the original arguments.

Step 3 — Verify

After gsd-fast completes (and scope remained ≤3 files throughout): run the verification command from gsd-fast output. If no automated verification is available, SB asks the user to confirm the change looks correct.

Step 4 — Confirm

SB displays a completion summary showing: change description, files modified count (confirmed ≤3), status: committed.

When NOT to use

Do not use silver:fast for:

  • Any change touching 4+ files
  • Business logic changes (even "small" ones)
  • Dependency additions or removals
  • Schema changes (database, API, config format)
  • Changes with downstream impact on other systems
  • Security-related changes

If in doubt, use /silver instead — it will run complexity triage and choose the right workflow.

Example invocation

/silver:fast Fix typo in README installation section

Silver Bullet confirms trivial scope (1 file, typo). Invokes gsd-fast. README updated, verified, committed. Done.
