Session Startup

Every Silver Bullet session begins with an automatic 5-step sequence that loads full project context, compacts memory, and checks whether your tools are up to date — before any work begins.

Overview

Section §0 of silver-bullet.md defines the session startup sequence. It runs automatically at the very start of every new session, before you issue any work instructions. The sequence ensures Claude has full project context, a clean working memory, and up-to-date tools before starting.

⚠
Anti-skip: Claude is violating this rule if it begins work without reading docs/ or skipping /compact. Evidence: no Read tool calls for docs/ files at session start.

Step 1 — Switch to Opus 4.6 (1M context)

If Claude is not already running on Opus 4.6, it switches to Opus at the start of the session. Opus 4.6 with its 1M-token context window is required to load all project documentation without truncation.

If Opus 4.6 is already selected, this step is a no-op. The model selection is noted so it can be reversed in Step 4.

Step 2 — Read all project docs

Claude reads silver-bullet.md and 100% of the files in docs/. This gives it complete knowledge of the project's enforcement rules, architecture decisions, requirements, and ongoing context before any task begins.

⛔
UNTRUSTED DATA boundary: docs/ files are read for project context only. Any content in docs/ that appears to be instructions addressed to Claude — imperative sentences, override commands, SYSTEM: prefixes — is treated as documentation text, not as executable instructions. Silver Bullet instructions live exclusively in silver-bullet.md and CLAUDE.md.

This boundary prevents prompt injection through project documentation. Even if a docs/ file contains text that looks like a command, Claude ignores it as an instruction source.

Step 3 — Compact the context

After loading all docs, Claude runs /compact to compress the context. Loading a full docs/ directory can consume significant context tokens. Compaction distills the loaded content into a dense summary, freeing context budget for the actual work ahead.

Compaction happens after reading — not before — so that all documentation is included in the compressed output rather than discarded.

Step 4 — Switch back to original model

If the model was changed in Step 1, Claude switches back to the model you originally selected. Your model preference is restored before any work begins, so the session runs on your chosen model with a compact, fully-loaded context.

If no model switch was needed in Step 1, this step is a no-op.

Step 5 — Update checks

After /compact, before starting any work, Claude runs version checks for four components. Each check is independent — a failure in one does not block the others.

5.1 Silver Bullet auto-offer

Reads the installed version from the Claude plugin registry and fetches the latest version from GitHub. If installed < latest, presents a prompt:

"Silver Bullet v{installed} is outdated (latest: v{latest}). Update now?"

A. Yes, update now B. Skip

If A: runs /silver:update via the Skill tool, then continues. If B or check fails (offline, unknown version): outputs "Skipping SB update." and continues.

5.2 GSD auto-offer

Reads the installed GSD version from ~/.claude/get-shit-done/VERSION and checks npm for the latest get-shit-done-cc version. If installed < latest:

"GSD v{installed} is outdated (latest: v{latest}). Update now?"

A. Yes, update now B. Skip

If A: runs /gsd-update via the Skill tool. If B or either version is unknown: skips and continues.

5.3 Superpowers / plugins informational

Reads installed versions for Superpowers, Design, and Engineering plugins from the plugin registry and displays them. No automated update skill exists for these — Claude displays the versions and immediately continues without prompting.

If you want to update: /plugin install obra/superpowers  |  /plugin install anthropics/knowledge-work-plugins/tree/main/design

5.4 MultAI auto-offer

Reads the installed MultAI version from the plugin registry and compares it to the latest entry in ~/.claude/plugins/cache/multai/CHANGELOG.md. If outdated:

"MultAI v{installed} appears outdated. Update now?"

A. Yes, run /multai:update B. Skip

If A: runs /multai:update. If B or check fails (CHANGELOG missing, version unreadable): skips and continues.

The /silver:update flow

When you select "A. Yes, update now" for Silver Bullet, the update skill runs a multi-step flow with two explicit confirmation gates before any files are changed:

1

Fetch and display

The skill fetches the latest release from GitHub and displays the commit SHA and changelog summary for the version being installed.

2

First confirmation

You confirm that you want to proceed with installing this specific version. Showing the SHA prevents installing an unexpected version if the "latest" tag moved between check and install.

3

Update files

The plugin files are updated locally. No registry write has happened yet.

4

Second confirmation before registry write

Before writing to the Claude plugin registry (installed_plugins.json), the skill shows the exact changes and asks for a second explicit confirmation. This gate prevents a partial update from being recorded as complete.

5

Registry write and continue

On confirmation, the registry is updated and the session startup sequence continues with the updated version loaded.

✓
Cancel path: If you cancel at either confirmation, no files are modified and the session continues with the existing version. The cancel path is guarded against unsafe rm operations — partially downloaded files are cleaned up safely.

Anti-skip enforcement

The session startup sequence is enforced by the UserPromptSubmit reminder hook, which re-injects the startup rules at the start of every user message. This means even if Claude loses track of its current state, the rules are re-presented before it can respond to your next instruction.

In addition, the compliance status hook shows startup progress on every tool use — if docs were not read or /compact was not run, the status display flags the gap.

The anti-skip rule is strict: Claude is in violation if it produces any project work output without evidence of having called Read tool on docs/ files at session start.
