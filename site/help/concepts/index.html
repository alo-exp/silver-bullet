Skills

A skill is a markdown file that contains a detailed prompt and instructions for Claude to follow when performing a specific task. When you invoke a skill, Claude reads the file and executes its instructions as if it were a step-by-step process guide.

How Skills Are Stored

Skills live in two locations:

  • ~/.claude/skills/ — flat .md files, one per skill
  • ~/.claude/plugins/cache/ — structured as publisher/plugin/version/skills/skill-name/SKILL.md

How Skills Are Invoked

Inside Claude Code, skills are invoked with the Skill tool using a slash-command shorthand:

/quality-gates # invokes the quality-gates skill /gsd:discuss-phase # invokes GSD's discuss-phase command /requesting-code-review # dispatches code-reviewer subagent for peer review /silver quality review # routes to the right skill automatically

The /silver Router

/silver is Silver Bullet's smart dispatcher. It accepts freeform natural language, matches your intent against the full routing table of SB skills, and invokes the right one — without you needing to know the exact skill name. For GSD planning and execution work, it delegates to /gsd:do automatically. It's the single entry point for users who don't know which skill to invoke.

Implicit coverage does not count. The enforcement hooks track actual Skill tool invocations. Saying "I already covered this while writing the code" is not valid — each skill must be explicitly invoked. This is intentional: the hooks can only measure what actually happened, not what Claude claims happened.

Skill Discovery

At the start of each task, Silver Bullet scans all installed skills and cross-references them against the task description. It surfaces candidates that might apply — for example, flagging /security when auth changes are involved. This prevents relevant skills from being forgotten.

Hooks

A hook is a shell script that Claude Code runs automatically in response to events — most importantly, after every tool use (PostToolUse). Silver Bullet uses hooks to enforce the workflow without relying on Claude's self-discipline.

Enforcement Hooks (4)

  • Skill tracker — Records which skills have been invoked this session. Used by the completion audit.
  • Stage enforcer — Hard stops if you try to advance to planning before discussion is complete.
  • Compliance status — Shows a progress bar of required steps completed on every tool use.
  • Completion audit — Blocks git commit, git push, and deploy commands if required skills are missing.

Support Hooks (4 + session start)

  • Semantic compression — Runs after each Skill invocation; TF-IDF-ranks your codebase and injects relevant context. See Semantic Context Compression.
  • Session log init — Creates the session log file on first Bash use; stores path in ~/.claude/.silver-bullet/session-log-path.
  • CI status check — Intercepts push/deploy Bash commands and verifies CI is green before allowing them to proceed.
  • Timeout check — Monitors for stall conditions and surfaces a timeout sentinel to break infinite loops.
  • Session start — Runs once at session open (on startup, /clear, or /compact); injects Superpowers and Design plugin context.
Hooks run outside Claude's context. They're shell scripts executed by Claude Code itself, not by Claude. This means they can't be overridden by Claude's reasoning or instructions — they're a hard system constraint.

Trivial Changes

Not every edit needs a full workflow. For small, non-logic changes (typos, copy fixes, config tweaks), you can bypass workflow enforcement with:

touch ~/.claude/.silver-bullet/trivial

This tells the hooks to skip enforcement for the next commit. Use it sparingly — it exists for genuine trivial changes, not as a shortcut for skipping review.

GSD — The Authoritative Orchestrator

GSD (Get Shit Done) is the planning and execution engine at the core of Silver Bullet. It manages the complete lifecycle of a development task — from requirements through shipment — using a structured file-based workflow in the .planning/ directory.

What GSD Manages

  • .planning/PROJECT.md — Project definition and goals
  • .planning/REQUIREMENTS.md — Authoritative requirements list (single source of truth)
  • .planning/ROADMAP.md — Phased implementation plan
  • .planning/{phase}-CONTEXT.md — Decisions captured in Discuss step
  • .planning/{phase}-PLAN.md — Detailed task-level implementation plan
  • .planning/{phase}-VERIFICATION.md — Goal-backward verification results

GSD Commands

GSD is invoked via slash commands that Claude Code resolves to GSD's installed workflows:

/gsd:new-project # Initialize project with requirements + roadmap /gsd:discuss-phase # Capture decisions for the current phase /gsd:plan-phase # Research + create task-level plan /gsd:execute-phase # Wave-based parallel execution /gsd:verify-work # Goal-backward verification /gsd:ship # Create PR with phase summaries /gsd:next # "Where am I?" — advance to next step /gsd:debug # Systematic debugging for any encountered bug

Superpowers

Superpowers is a design and review plugin that provides skills for brainstorming, system design, code review, and UX copy. In the Silver Bullet workflow, Superpowers fills the design and review role while GSD handles planning and execution.

Key Superpowers Skills Used in Silver Bullet

  • /brainstorming — Explores problem space before planning begins
  • /system-design — Architecture decisions for new services or components
  • /design-system — UI component and design token decisions (UI work only)
  • /ux-copy — User-facing text and messaging decisions (UI work only)
  • /code-review — Structured peer code quality review: security (SQL injection, XSS, auth flaws), performance (N+1, unbounded queries), correctness (edge cases, race conditions), and maintainability. Run before dispatching the automated reviewer. From Engineering plugin.
  • /requesting-code-review — Dispatches superpowers:code-reviewer for peer code quality review (security, performance, readability). Loop until 2 consecutive approvals.
  • /receiving-code-review — Triage all feedback: accept or reject each item with reasoning before fixing

GSD / Superpowers Ownership Model

Both GSD and Superpowers are capable of producing plans, specifications, and executing tasks. Silver Bullet enforces a strict ownership split to prevent duplication and conflicts:

ConcernOwnerRule
RequirementsGSD.planning/REQUIREMENTS.md is the single source of truth. Superpowers must not maintain a separate requirements list.
PlanningGSDUse /gsd:plan-phase for all plans. When Superpowers' brainstorming offers to hand off to writing-plans, redirect to /gsd:plan-phase.
ExecutionGSDAlways use /gsd:execute-phase (wave-based). Never use superpowers:subagent-driven-development for project work.
Design specsSuperpowersSave to docs/specs/YYYY-MM-DD-<topic>-design.md. The default Superpowers path (docs/superpowers/specs/) is not used.
Code reviewSuperpowers + Engineering/code-review (Engineering), /requesting-code-review, /receiving-code-review, and superpowers:code-reviewer are for review only, never execution.
Never use subagent-driven-development for project work. This Superpowers skill is powerful but operates outside GSD's enforcement model. Using it for execution bypasses quality gates, atomic commits, and verification requirements.

Quality Gates

Quality gates are a mandatory pre-planning step where Claude evaluates the proposed design against 8 quality dimensions. All must pass before planning begins — a single is a hard stop.

Modularity
Components are independently replaceable
Reusability
Components shared without modification
Scalability
Handles growth without redesign
Security
Defense-in-depth, least privilege
Reliability
Graceful failures, fault tolerance
Usability
Intuitive APIs and interfaces
Testability
Isolated, fast, deterministic tests
Extensibility
Open for extension, closed for modification

How Quality Gates Are Evaluated

The /quality-gates skill dispatches 8 parallel agents — one per dimension — each in an isolated worktree. Results are synthesised by Claude. The most conservative finding wins: if one agent flags a dimension as , the whole gate fails.

DevOps Quality Gates

For infrastructure work, /devops-quality-gates applies 7 IaC-adapted dimensions (usability is excluded — it's application-specific) — focusing on idempotency, drift prevention, least-privilege IAM, and infrastructure reliability instead of application concerns.

Enforcement Layers

Ten independent enforcement layers — Silver Bullet hooks covering every hook type (PreToolUse, PostToolUse, Stop, UserPromptSubmit, SubagentStop), GSD runtime hooks, plus redundant instructions that reinforce the same rules across multiple files:

1

Pre+PostToolUse — Skill Tracker

Records every Skill tool invocation in the session state file. The source of truth for which required steps have been completed.

2

Pre+PostToolUse — Stage Enforcer

Reads session state and issues a HARD STOP if Claude attempts to advance to the next phase before completing the current one (e.g., starting PLAN before /quality-gates passes).

3

Pre+PostToolUse — Compliance Status

After every tool use, outputs a progress indicator showing which required steps are complete and which are pending. Keeps you informed without being obtrusive.

4

Pre+PostToolUse — Completion Audit

Intercepts git commit, git push, and deploy commands. Uses a two-tier check: commits and pushes only require quality gates to have run; PR creation, deploys, and releases require the full required skill list. Blocks with a specific list of what's incomplete. Registers as PreToolUse so blocking is unbypassable — even in bypass-permissions mode.

5

PreToolUse — CI Gate

Intercepts git push commands and checks the most recent CI run for the branch. Broken builds cannot be pushed — CI must be green first.

6

Stop / SubagentStop — Completion Gate

Fires when Claude declares a task complete. Blocks if required skills are still missing. Re-injected on every message so it survives compaction and cannot be lost mid-session.

7

UserPromptSubmit — Prompt Reminder

Before every user message is processed, re-injects the list of missing required skills and the core enforcement rules. Ensures Claude cannot "forget" the workflow even after context compaction.

8

PreToolUse — Forbidden Skill Gate

Blocks deprecated or project-forbidden skills before they execute. Configurable via skills.forbidden in .silver-bullet.json.

9

GSD Workflow Guard

GSD’s own hook detects file edits made outside a /gsd:* command and warns immediately. Independent of Silver Bullet’s hooks.

10

Redundant Instructions + Anti-Rationalization

The workflow is described in both the workflow file and CLAUDE.md. Explicit rules prohibit Claude from deciding a step is “not applicable” without user approval, or claiming it “already covered” a skill without actually invoking it.

Session Modes

Every session begins with a mode selection that controls how much Claude pauses for approval:

Interactive (default)

Claude pauses at every phase gate. You review and approve each stage. Best for learning the workflow or working on sensitive/complex tasks where human oversight matters at each step.

Autonomous

Claude drives start-to-finish. Decisions are logged. Genuine blockers are queued and surfaced at the end. Best for well-defined tasks when you want maximum throughput.

The mode is written to ~/.claude/.silver-bullet/mode and read by hooks throughout the session. If the file is missing or unreadable at any point, the system defaults to Interactive.

Autonomous Mode Safeguards

Even in autonomous mode, Silver Bullet has stall-detection to prevent infinite loops. Two independent tiers run in parallel:

  • Wall-clock sentinel: If no skill has been recorded in the last 10 minutes, a timeout warning fires.
  • Call-count tier: Warns at 30, 60, and 100+ tool calls since the last skill invocation — escalating in urgency at each threshold.
  • Identical-call guard: The same tool call with identical arguments producing the same result 2+ times consecutively is treated as a stall.

On any stall: Claude makes a best-judgment call, moves on, and logs the decision for review.

Agent Teams

Silver Bullet uses Claude's sub-agent capability to parallelize work. When a phase has independent implementation units (different modules, different quality dimensions), multiple agents run simultaneously in isolated git worktrees.

How It Works

  1. GSD's execute-phase identifies independent tasks within a wave
  2. Each task is dispatched as a separate agent with isolation: "worktree"
  3. All agents run in parallel (with run_in_background: true in Autonomous mode)
  4. After all agents complete, a merge gate resolves any conflicts before the next wave starts
Worktree isolation means each agent works on its own copy of the repository. Changes from one agent can't interfere with another mid-flight — they're only merged once the merge gate runs.

Semantic Context Compression

Silver Bullet includes an automatic context optimization system that runs as a PostToolUse hook. It scans your source and documentation files, ranks chunks by relevance to the current phase goal using TF-IDF scoring, and injects the most relevant context into Claude's working memory before each tool use.

This keeps Claude focused on the right parts of a large codebase without exhausting its context window on irrelevant files. Results are cached between tool uses and invalidated when files change.

Configure it in .silver-bullet.json under "semantic_compression".

Standard File Structure

After /silver:init initializes a project, you'll find this structure:

# Project root CLAUDE.md # Active workflow + Silver Bullet rules .silver-bullet.json # Project config (semantic compression, etc.) .github/workflows/ci.yml # CI pipeline (auto-generated if absent) # Planning (managed by GSD) .planning/ PROJECT.md # Project goals + metadata REQUIREMENTS.md # Authoritative requirements ROADMAP.md # Phase breakdown {phase}-CONTEXT.md # Per-phase decisions {phase}-PLAN.md # Per-phase task plan {phase}-VERIFICATION.md # Verification results .context-cache/ # Semantic compression cache # Documentation (maintained by Silver Bullet) docs/ PRD-Overview.md # High-level requirements (syncs from .planning/) Architecture-and-Design.md # High-level arch + principles only Testing-Strategy-and-Plan.md CICD.md KNOWLEDGE.md # Architectural decisions, gotchas, patterns CHANGELOG.md # Task log (newest first) specs/ # Detailed design specs (Superpowers) workflows/ # Active workflow files sessions/ # Per-session logs

Routing Logic

/silver is the single entry point for all Silver Bullet work. Every message is classified before routing — using complexity triage (Trivial / Simple / Fuzzy / Complex) and a full intent-matching table — so the right workflow always fires. The §2g bare instruction interception rule means you don't need to invoke /silver explicitly: typing a plain work request triggers it automatically.

Routing Logic →

Complexity triage decision tree, full 7-workflow routing table, §2g bare instruction interception, and ship disambiguation.

Complexity Triage

Trivial → silver:fast. Fuzzy → silver:explore first. Simple → direct workflow. Complex → explore + brainstorm. Prevents over-engineering trivial changes.

Verification Before Completion

gsd-verify-work is a non-skippable gate that runs before any workflow completes. It is a §3 rule — it cannot be waived via §10 preferences or the step-skip protocol. "Done" means tests pass and behavior is confirmed, not just that code was written.

Verification Before Completion →

What gsd-verify-work checks, when it fires in each workflow, what counts as verified, and what cannot be skipped.

Non-Skippable Gates

Three gates are permanently non-skippable: silver:security, silver:quality-gates pre-ship, and gsd-verify-work. No §10 preference can waive them.

User Workflow Preferences

§10 of silver-bullet.md stores per-project routing preferences — permanent step skips, lightweight alternative substitutions, and workflow customizations recorded after using the step-skip protocol. Changes are always written to both silver-bullet.md and templates/silver-bullet.md.base atomically.

User Workflow Preferences →

How §10 routing preferences work, the step-skip protocol, which gates can and cannot be permanently waived, and how preferences are recorded.

Step-Skip Protocol

When asked to skip a step, Silver Bullet explains why it exists, offers lettered options (skip / lightweight alternative / show current state), and records permanent decisions in §10 only after explicit confirmation.

Session Startup

At the start of every new session, Silver Bullet automatically performs five steps: switch to Opus 4.6 (1M context), read all project docs, compact the context, switch back to the original model, and check for updates to Silver Bullet, GSD, plugins, and MultAI. This is the §0 startup sequence.

The full §0 5-step automatic initialization sequence: model switching, doc loading, context compaction, and version checks for all four components.

Automatic Updates

Silver Bullet, GSD, Superpowers/Design/Engineering plugins, and MultAI are each version-checked at session start. Outdated components surface an update prompt before work begins.
