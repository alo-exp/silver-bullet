Routing Logic

How /silver classifies every instruction, performs complexity triage, and dispatches to the right orchestration workflow — automatically.

Overview

/silver is the single entry point for all Silver Bullet work. You never invoke orchestration workflows directly — you either type a bare instruction or call /silver explicitly, and the router handles classification and dispatch.

Every message passing through /silver goes through two sequential steps before work begins:

  1. Bare instruction interception (§2g) — if your message is a plain work request rather than a slash command, /silver is invoked automatically before anything else happens.
  2. Complexity triage — the router classifies the request as Trivial, Simple, Fuzzy, or Complex to select the right workflow and determine whether an exploration step is needed first.

This design prevents two common failure modes: accidentally skipping orchestration by typing work requests directly, and over-engineering trivial changes with a full workflow pipeline.

Bare Instruction Interception (§2g)

Silver Bullet intercepts bare natural-language instructions at the start of every user turn, before any tool use. When you type a plain work request — not a slash command — /silver is invoked automatically.

Why interception exists

Without interception, it's easy to accidentally bypass orchestration. If you type "Add a login form" directly, a plain AI assistant would start writing code immediately, skipping quality gates, brainstorming, planning, and all verification steps. Interception makes /silver the mandatory gateway for all non-trivial work.

How it works

  1. User sends a message that is not a slash command
  2. Silver Bullet checks: is this a non-trivial work request?
  3. If yes: intercept — invoke /silver via the Skill tool, passing the original instruction as arguments
  4. /silver performs complexity triage and routes to the correct workflow
  5. The work proceeds through the full orchestrated pipeline

What gets intercepted vs. exempted

Message typeIntercepted?Example

Anti-skip rule: Silver Bullet violates its own enforcement design if it reads a non-trivial bare instruction and begins responding or executing work without first invoking /silver. The router exists to ensure every task reaches the right skill — bypassing it defeats the enforcement model.

Complexity Triage Decision Tree

Once a message reaches /silver (either via interception or direct invocation), the router classifies it using a 4-way triage. The classification determines which workflow fires and whether a scoping step runs first.

The four classifications

Work through these questions in order:

Is the change ≤3 files? Is it a typo, rename, or config tweak? YES → Trivial → route to silver:fast (no planning overhead) Is the intent vague or scope unclear? Would you need to explore before planning? YES → Fuzzy → run silver:explore first, then re-classify Is the scope clear and does it fit within a single phase? YES → Simple → route to the correct workflow, skip explore step Multi-phase, cross-cutting, architectural, or has significant unknowns? YES → Complex → full workflow with silver:explore + silver:brainstorm

Classification summary

ClassificationCharacteristicsRouteExplore step?
Trivial ≤3 files, typos, renames, config tweaks, one-liners silver:fast No
Simple Clear scope, single phase, well-understood problem Intent-matched workflow No — skip to workflow
Fuzzy Vague intent, unclear scope, needs clarification silver:explore → re-classify Yes — explore first
Complex Multi-phase, cross-cutting, architectural, significant unknowns Intent-matched workflow with full pre-work Yes — explore + brainstorm
Trivial changes bypass enforcement. silver:fast routes directly to gsd-fast with no planning overhead. If the work expands beyond 3 files during execution, Silver Bullet stops and re-routes through /silver for proper classification. §10 routing preferences are not applied to silver:fast by design.

Full Routing Table

After complexity triage confirms the request is not Trivial or Fuzzy, /silver matches intent signals to one of seven orchestration workflows:

WorkflowEntry triggersFirst step
/silver:feature "add X", "build X", "implement X", "new feature", "enhance X", "extend X" silver:intel → product-brainstorming → silver:brainstorm
/silver:bugfix "bug", "broken", "crash", "error", "regression", "failing test", "not working" SB triage → systematic-debugging → gsd-debug
/silver:ui "UI", "frontend", "component", "screen", "design", "interface", "page", "layout" silver:intel → product-brainstorming → silver:brainstorm → gsd-ui-phase
/silver:devops "infra", "CI/CD", "deploy", "pipeline", "terraform", "IaC", "kubernetes", "cloud", "ops" silver:intel → silver:blast-radius → silver:devops-skill-router
/silver:research "how should we", "which technology", "compare X vs Y", "spike", "investigate", "architecture decision" silver:explore → MultAI research → silver:brainstorm
/silver:release "release", "publish", "version", "go live", "cut a release", "tag v", "ship to users" silver:quality-gates → gsd-audit-uat → gsd-audit-milestone
/silver:fast "trivial", "quick fix", "typo", "one-liner", "config value", or complexity triage → Trivial Confirm ≤3 files → gsd-fast

Ship Disambiguation

The word "ship" is ambiguous in Silver Bullet. It can mean a phase-level merge (a branch PR) or a milestone-level publish (a versioned release). /silver disambiguates at routing time based on signal keywords:

Signal in messageRouteWhat it does
Contains version number (v2.0, 1.4.0, etc.) silver:release Milestone publish: quality gates, audit, create GitHub Release, git tag
Contains "changelog" or "release notes" silver:release Milestone publish
Contains "go live", "to production", "publicly" silver:release Milestone publish
Active phase in progress, no version signal gsd-ship Phase-level merge: push branch, create PR
No active phase, end of milestone silver:release Milestone publish
These are different scopes. gsd-ship runs inside any workflow as the final phase-level step (push → PR). silver:release is a dedicated milestone-level workflow that runs quality gates, UAT audit, milestone audit, and then creates a versioned GitHub Release. Running the wrong one at the wrong scope is a common mistake — let /silver disambiguate for you.
