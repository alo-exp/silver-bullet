Composable Workflow Orchestration

How /silver replaces fixed pipelines with dynamic path composition — classifying context, selecting from 18 paths, and supervising execution via WORKFLOW.md.

What are Composable Paths?

Silver Bullet's composable paths architecture replaces fixed step-by-step pipelines with a catalog of 18 named paths. When you invoke /silver, it classifies your context and dynamically composes a chain of paths appropriate to the work — no two compositions need be identical.

Each path is a discrete unit with a contract: prerequisites, trigger signals, steps, produced artifacts, review cycle, GSD state impact, and exit condition. Paths compose into chains. Chains execute under supervision.

Why composition instead of fixed pipelines? A feature composition includes PLAN, EXECUTE, VERIFY, REVIEW, SECURE, SHIP. A debug composition inserts DEBUG before EXECUTE. A release composition includes RELEASE but not EXECUTE. Fixed pipelines cannot express this variability cleanly — composable paths can.

The 18 Paths

Every path has a number, a name, and a one-line description. The trigger column describes what causes /silver to include this path in a composition.

How Composition Works

When you invoke /silver, composition happens in four steps:

  1. Context classification — /silver reads the request, current STATE.md position, and file context to determine workflow type (feature, bugfix, UI, devops, research, release, trivial).
  2. Path selection — Based on classification, /silver selects which of the 18 paths apply. Always-included paths (PATH 7 EXECUTE, PATH 11 VERIFY, PATH 12 QUALITY GATE, PATH 13 SHIP) are anchors. Context-dependent paths (PATH 2 EXPLORE, PATH 6 DESIGN CONTRACT, PATH 8 UI QUALITY, PATH 14 DEBUG) are added based on signals.
  3. Chain ordering — Paths are ordered by their dependency graph. PATH 0 before PATH 1 before PATH 5 before PATH 7, etc. Dynamic insertions (e.g. PATH 14 DEBUG on failure) are injected at the appropriate point in the running chain.
  4. Proposal — /silver proposes the composed chain to the user, explaining which paths are included and why, before execution begins.
Example composition for a feature request: PATH 1 (ORIENT) → PATH 12 (QUALITY GATE, pre-plan) → PATH 5 (PLAN) → PATH 7 (EXECUTE) → PATH 11 (VERIFY) → PATH 9 (REVIEW) → PATH 10 (SECURE) → PATH 12 (QUALITY GATE, pre-ship) → PATH 13 (SHIP) → PATH 16 (DOCUMENT). PATH 14 (DEBUG) is available to insert dynamically if PATH 7 fails.

WORKFLOW.md Tracking

WORKFLOW.md is the real-time state tracking artifact for an active composition. It is created when a composition begins and updated after every path completes.

What WORKFLOW.md Contains

  • Active path — which path is currently executing
  • Completed paths — log of paths that have exited successfully, with timestamps
  • Deferred improvements — items noted during execution that are not blocking but should be addressed later
  • Dynamic insertions — paths inserted at runtime (e.g. PATH 14 DEBUG after a failure)
  • Composition plan — the full ordered chain proposed at composition time

Why WORKFLOW.md Matters

WORKFLOW.md gives both the user and the agent a shared, persistent view of what is happening. If a session is interrupted, WORKFLOW.md allows the next session to resume from the correct point in the chain without re-running completed paths. It also provides an audit trail for post-hoc review.

Supervision Loop

Each path in a composition runs under a supervision loop:

  1. Pre-path check — Prerequisites are verified before the path begins. If prerequisites are missing, the path is blocked and the user is notified.
  2. Step execution — The path's steps execute in order, with mandatory steps always running and as-needed steps gated on context signals.
  3. Exit verification — The path's exit condition is checked. If not met, the supervision loop either retries or inserts a corrective path (e.g. PATH 14 DEBUG on execution failure).
  4. Stall detection — If a path makes no progress for a configured timeout, a stall sentinel is surfaced to break the loop.
  5. Progress reporting — On every tool use, the compliance status hook shows the current path and overall chain progress.
PATH 11 VERIFY is NON-SKIPPABLE. The supervision loop cannot exit the chain before PATH 11 completes. Even if the user requests to skip it, the enforcement hooks block any git commit or git push until verification has run and passed.
