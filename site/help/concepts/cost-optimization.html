Automatic Cost Optimization

Silver Bullet automatically routes each AI agent to the right model tier — Opus for judgment-heavy work, Sonnet for execution, Haiku for structured output — with zero manual switching.

How It Works

Every GSD workflow spawns multiple specialized subagents to complete a phase — a planner, one or more executors, a verifier, sometimes a researcher and a code reviewer. Without cost optimization, every agent would use the same model, paying Opus prices for tasks that need nothing more than Haiku.

Silver Bullet solves this automatically. When a workflow spawns a subagent, the model routing system looks up that agent type in a profile table and selects the appropriate model tier — before the agent even starts. No prompting, no switching, no configuration required at runtime.

Request flow
/silver → /gsd:plan-phase → spawns gsd-planner → resolve-model → Opus → spawns gsd-executor → resolve-model → Sonnet → spawns gsd-verifier → resolve-model → Opus → spawns gsd-codebase-mapper → resolve-model → Haiku

The routing system reads .planning/config.json for the active model_profile, looks up the agent in the profile table, and passes the resolved model to the Task() call. All 24 GSD agent types are covered.

Zero friction. You never choose a model mid-session for GSD work. The routing table encodes the right decision for every agent type. Just run your workflows normally.

Tier Rationale

The three tiers reflect a single principle: spend on reasoning where reasoning changes the outcome.

Opus — Design, Review & Verification

Used where judgment quality directly impacts the result. A weak plan wastes an entire execution phase. A missed security issue ships to production. A false-positive verification passes broken code. These are the moments where Opus earns its cost.

  • Design agents make architecture decisions and produce plans that executors follow verbatim. Bad decomposition here cascades into every downstream task.
  • Review agents catch bugs, security issues, and plan quality problems. Sonnet can follow patterns; catching subtle security flaws requires genuine reasoning.
  • Verification agents check whether the code actually delivers what the phase promised — goal-backward reasoning, not pattern matching.

Sonnet — Execution & Research

Used for agents that follow explicit instructions. Executors implement a PLAN.md — the reasoning was already done by the planner. Researchers fetch and summarize information. Doc writers fill structured templates. These tasks need competence, not depth.

Haiku — Structured Output

Used for agents that extract and format, not reason. Codebase mapping reads files and produces structured summaries. Intel updating writes formatted analysis files. User profiling scores behavioral dimensions from session data. The output format is well-defined; the task is extraction.

Inline Skills

Not all Silver Bullet capabilities run as GSD subagents. Skills from Anthropic's plugin suite (design, engineering, product-management, marketing), MultAI, Context7, and Superpowers run inline — they execute in the current session context rather than as spawned agents.

Inline skills use the session model — whatever model your Claude Code session is currently set to. The GSD model routing table does not apply to them.

Default session model is Sonnet. Silver Bullet sets claude-sonnet-4-6 as the session default. Inline skills run on Sonnet automatically. If you need deeper reasoning for a specific design or architecture skill, switch to Opus with /model claude-opus-4-6 before invoking it, then switch back.

Why inline skills aren't routed automatically

Inline skills are conversational — they ask clarifying questions, present options, and iterate with you mid-execution. Subagents are fire-and-forget. Converting interactive skills to subagents would break the back-and-forth that makes them useful, and require serializing the full conversation context into every spawn. The trade-off isn't worth it.

Model Profiles

The routing table has four built-in profiles. Switch between them with /gsd-set-profile <name>.

ProfilePhilosophyBest for
balanced Opus for design/review/verification; Sonnet for execution/research; Haiku for structured output Default — normal development work
quality Opus for all agents that make decisions; Sonnet for structured-output agents Critical architecture, high-stakes releases
budget Sonnet for anything that writes code; Haiku for research and verification High-volume work, conserving quota
adaptive Opus for planning and debugging; Haiku for checkers and auditors Optimizing cost without sacrificing plan quality
inherit All agents use the session model Non-Anthropic providers, OpenRouter, local models

Project Setup

Model routing is activated automatically when you initialize a project with /silver-init. The init skill runs /gsd-new-project (if not already done) and verifies that model_profile: "balanced" is set in .planning/config.json.

To verify your project is correctly configured:

# Check active profile node "$HOME/.claude/get-shit-done/bin/gsd-tools.cjs" config-get model_profile # Should output: balanced

If you see legacy keys (planner_model, researcher_model, checker_model) in your config instead of model_profile, your project was initialized before this feature was introduced. Migrate with:

# Apply the balanced profile (removes legacy keys automatically) /gsd-set-profile balanced

Per-Agent Overrides

You can override any individual agent without changing the overall profile. Add model_overrides to .planning/config.json:

{ "model_profile": "balanced", "model_overrides": { "gsd-executor": "opus", "gsd-codebase-mapper": "sonnet" } }

Overrides take precedence over the profile for that specific agent. Valid values: opus, sonnet, haiku, inherit, or any fully-qualified model ID (e.g. "claude-opus-4-6[1m]" for the 1M context variant).

Override sparingly. The balanced profile was designed to minimize cost without compromising outcome quality. Upgrading low-tier agents to Opus usually adds cost without measurable benefit. The most common legitimate override is bumping gsd-debugger to claude-opus-4-6[1m] for sessions involving very large codebases.
