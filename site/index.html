The Problem

AI Agents Skip Steps — And Lose Context

Even when your CLAUDE.md explicitly says “always plan first,” Claude rationalizes its way past it. And as context fills, quality silently degrades.

Skipped Planning

Claude jumps straight to coding. No brainstorming, no spec, no design review. The path of least resistance is always to just start writing code.

No Quality Gates

Security, scalability, and reliability are afterthoughts — not enforced checkpoints. Modularity and testability are never considered before implementation.

Premature Completion

Work is declared “done” without code review, testing strategy, or verification. Commits happen before the workflow is complete.

Context Rot

As the context window fills, output quality silently degrades. The best workflows in the world mean nothing if the agent is operating on a rotting context.

The root cause: Great workflows exist. Great enforcement tools exist. Great execution engines exist. But they’ve never been combined into a single, coherent, enforced system — until Silver Bullet.

Spec-Driven Development

Build From Spec, Not Assumption

Silver Bullet’s spec pipeline takes you from raw requirements to a validated, traceable SPEC.md before a single line of code is written. External sources ingested via MCP. Pre-build validation enforced by hooks.

Spec Creation silver:spec

AI-driven spec elicitation with canonical SPEC.md format. Structured requirements, acceptance criteria, assumption tracking, and source input cross-reference built in from the start.

External Ingestion silver:ingest

Ingest specs from JIRA, Figma, and Google Docs via MCP connectors. No custom API code — all external data access delegated to MCP. Produces a structured INGESTION_MANIFEST.

Pre-Build Validation silver:validate

Validate specs before planning begins. Spec floor enforcement, PR-to-spec traceability, and UAT pipeline gate. Broken specs are blocked before they become broken builds.

Multi-Repo Coordination

Cross-repo spec fetch via GitHub CLI. Read-only cache written to .planning/SPEC.main.md. No git submodules, no coupling — just a clean reference for cross-repo alignment.

Artifact Review System

Every Artifact Reviewed. Every Time.

Eight domain-specific reviewer skills enforce quality across the full planning artifact chain. Two consecutive clean passes required before approval. No exceptions.

8 Artifact Reviewers

SPEC, DESIGN, REQUIREMENTS, ROADMAP, CONTEXT, RESEARCH, INGESTION_MANIFEST, and UAT — each reviewer runs domain-specific QC checks tailored to its artifact type.

2-Consecutive-Clean-Pass Enforcement

Every artifact must pass two consecutive clean review rounds before it is approved. State is tracked per artifact using SHA256-keyed JSON — no gaming the system.

Configurable Review Depth

Configure deep, standard, or quick review depth per artifact type in .planning/config.json. Standard is the default for backward compatibility.

Cross-Artifact Consistency

Validates SPEC-to-REQUIREMENTS-to-ROADMAP-to-DESIGN alignment in a single pass. Wired into milestone completion and the release workflow — misaligned artifacts block shipping.

Review Analytics

Measure Your Review Health

Every review round emits structured metrics. Silver Bullet gives you data-driven visibility into artifact quality trends, round counts, and finding categories — right from the CLI.

JSON Lines Metrics

Every review round emits structured metrics to a JSON Lines file. Pass rates, rounds to clean pass, and finding categories — all captured automatically with no extra configuration.

silver-review-stats

Run silver-review-stats to generate summary reports from review metrics. Three report tables cover pass rates, rounds to clean pass, and finding categories by artifact type.

Verification-Before-Completion

The review loop enforces that verification completes before any completion claim. No artifact can be declared done without confirmed verification in the review record.

Cost Optimization

Right Model, Right Task, Right Cost

Silver Bullet automatically routes each sub-agent to the optimal model tier. Sonnet (low thinking effort) is the default for 22 of 24 agents — fast, accurate, low cost. Opus is reserved for the two tasks where deeper reasoning genuinely matters: planning and security auditing.

Sonnet: Default (22 agents)

Execution, research, documentation, verification, code review, testing, codebase mapping, and more. Sonnet handles 22 of 24 agents — fast and accurate without unnecessary overhead.

Opus: Reserved (2 agents)

Only gsd-planner (architectural reasoning, MECE decomposition) and gsd-security-auditor (adversarial threat modeling) run on Opus — where reasoning depth directly determines outcome quality.

60–80% Cost Reduction

Running everything on Opus is like using a sports car for grocery runs. Sonnet-default routing gives each task the right engine — and keeps your token bill from spiraling.

Automatic, not manual. Model assignments are baked into each agent definition — model: sonnet or model: opus in YAML frontmatter. No configuration needed. The routing is built into the system and overridable per-agent if you need to adjust.

Architecture

Ten Enforcement Layers

Skill tracker, stage enforcer, compliance status, completion audit, CI gate, session management, Stop hook, UserPromptSubmit reminder, forbidden-skill gate, and redundant instruction enforcement. Each layer is independently sufficient. Together, they make bypassing the workflow extremely difficult to do by accident.

1

Skill Tracker

A PostToolUse hook fires on every Skill invocation, recording completed steps to a state file. This is the ground truth that every other enforcement layer reads from.

2

Stage Enforcer

A PostToolUse hook fires on every file edit. HARD STOP if quality gates are incomplete. Enforces strict phase ordering: DISCUSS → QUALITY GATES → PLAN → EXECUTE → VERIFY.

3

Compliance Status

Every tool use triggers a progress display: current phase, completed steps, total score, and the next required action. Compliance is always visible — there’s no hiding from it.

4

Completion Audit

A PostToolUse hook intercepts git commit, push, and deploy commands. If the full workflow isn’t complete, the operation is blocked with a list of remaining required steps.

5

CI Status Check

A PreToolUse hook intercepts git push commands and verifies that the most recent CI run for the branch is green before allowing the push. Broken builds cannot be pushed.

6

Session Management

A SessionStart hook tracks the current git branch. When the branch changes between sessions, the full skill-recording state resets — preventing skills from Feature A from satisfying enforcement for Feature B.

7

Stop Hook

A Stop/SubagentStop hook fires when Claude tries to declare a task complete. If any required skills are missing from the state file, the declaration is blocked with a list of what still needs to run.

8

Prompt Reminder

A UserPromptSubmit hook re-injects the missing skills list and core enforcement rules before every user message — ensuring each new prompt starts with full context of what is still required.

9

Forbidden Skill Gate

A PreToolUse hook intercepts every Skill invocation. Deprecated or project-forbidden skills are blocked before they execute, with a clear message pointing to the correct replacement.

10

Redundant Instructions + Anti-Rationalization

The same rules appear in three places: CLAUDE.md, the workflow file, and every hook message. Explicit language names and blocks common excuses: “This is simple enough to skip,” “I’ll do the review mentally.” Named = blocked.

Quality Gates

Quality Dimensions. Enforced Checkpoints.

The full-dev-cycle applies 9 quality dimensions via /quality-gates. The devops-cycle applies 7 IaC-adapted dimensions via /devops-quality-gates (usability and AI/LLM safety excluded) plus a mandatory /blast-radius assessment. Every dimension is a hard stop — not a warning.

Modularity

Small, focused modules. File size limits. Change locality max 3–5 files. Single responsibility enforced.

/quality-gates

Reusability

DRY, composable components. Rule of Three. No duplication across the codebase.

/quality-gates

Scalability

Stateless by default. Indexed queries. Async processing. Resource limits defined upfront.

/quality-gates

Security

OWASP Top 10. Input validation. Secrets management. Defense in depth. No trust boundaries crossed. SENTINEL hardening: shell injection fixes and markdown injection prevention built in.

/quality-gates

Reliability

Fault tolerance. Retries with backoff. Circuit breakers. Graceful degradation under load.

/quality-gates

Usability

Intuitive APIs. Actionable error messages. Accessibility. Progressive disclosure of complexity.

/quality-gates

Testability

Dependency injection. Pure functions. Test seams. Deterministic behavior. No hidden state.

/quality-gates

Extensibility

Open-closed principle. Plugin architecture. Versioned interfaces. Change without breakage.

/quality-gates

DevOps Cycle: Blast Radius First. Before quality gates, the devops-cycle requires a /blast-radius assessment — mapping change scope, downstream dependencies, failure scenarios, and rollback plans. Changes rated HIGH require explicit approval and a runbook. CRITICAL changes are a hard stop requiring CAB review.

Installation

Get Started in Two Minutes

Install Silver Bullet and initialize your project. Dependency plugins are guided automatically during setup.

# Install Silver Bullet
/plugin install alo-exp/silver-bullet
 
# Initialize your project
/silver:init
# Guides you through dependencies, then asks: app or DevOps project?
 
# Use /silver as your single entry point
/silver quality review my architecture
# Routes to the right skill automatically

Enforcement starts immediately. After running /silver:init, hooks are active and the workflow is live. Use /silver <anything> as your single entry point — it routes to the right skill automatically, or delegates to /gsd:do for planning and execution work. For app projects, the 20-step full-dev-cycle activates. For DevOps projects, the 24-step devops-cycle activates with blast radius assessment, IaC-adapted quality gates, and automatic detection of installed DevOps plugins. Missing dependencies are detected and guided during /silver:init — no manual setup required.
