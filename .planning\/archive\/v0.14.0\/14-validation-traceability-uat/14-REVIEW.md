# Phase 14: Code Review Report

**Reviewed:** 2026-04-09
**Depth:** standard
**Files Reviewed:** 6
**Status:** issues_found

## Summary

Phase 14 delivers three hooks (spec-session-record.sh, pr-traceability.sh, uat-gate.sh), two SKILL.md updates, and a template documentation section. The overall architecture is sound and the security mitigations cited in the threat model (umask 0077, append-only PR body, jq-based skill extraction) are correctly implemented.

The most serious issue is in `pr-traceability.sh`: raw WARN findings read from VALIDATION.md are interpolated directly into a shell heredoc without any escaping, which allows a maliciously crafted VALIDATION.md (or a finding text containing backticks or `$(...)`) to inject commands into the constructed string that is later written to a temp file and passed to `gh pr edit`. Additionally, a hardcoded `/opt/homebrew/bin/gh` path will silently do nothing on Linux or non-Homebrew macOS installs. Several edge cases in the UAT gate and the silver-validate SKILL.md also warrant attention.

## Critical Issues

### CR-01: Command injection via unescaped WARN findings in PR body construction

**File:** `hooks/pr-traceability.sh:60-71`

**Issue:** `warn_items` is populated directly from `grep 'FINDING [WARN]' .planning/VALIDATION.md` (line 45) and then embedded unquoted inside a heredoc (line 69: `${warn_items:-None}`). Shell heredocs do perform parameter expansion, so any content in `warn_items` that contains `$(...)` or backtick expressions will be evaluated by the shell at heredoc expansion time. A FINDING text written by an attacker who controls a SPEC.md assumption (e.g., `[ASSUMPTION: $(rm -rf ~)]`) would be faithfully echoed into VALIDATION.md by silver-validate and then executed here.

**Fix:** Assign the traceability block after sanitising `warn_items`, or use `printf '%s'` with a variable that is never expanded inside the heredoc:

```bash
# Instead of embedding $warn_items inside a heredoc (which expands it),
# write the static skeleton to the temp file and then append warn_items with printf.
{
  printf '\n---\n## Spec Traceability (auto-generated by Silver Bullet)\n'
  printf -- '- Spec: .planning/SPEC.md (v%s)\n' "${spec_version:-unknown}"
  printf -- '- JIRA: %s\n' "${jira_id:-n/a}"
  printf -- '- Requirements covered: see SPEC.md ## Acceptance Criteria\n'
  printf '\n### Deferred items (WARN findings from silver-validate)\n'
  printf '%s\n' "${warn_items:-None}"
} >> "$tmpfile"
```

This ensures `warn_items` is treated as data (passed to `printf %s`), not as shell code.

## Warnings

### WR-01: Hardcoded `/opt/homebrew/bin/gh` path breaks non-macOS / non-Homebrew installs

**File:** `hooks/pr-traceability.sh:49,57,79`

**Issue:** `gh` is invoked via the absolute path `/opt/homebrew/bin/gh` in three places. On Linux, or on macOS where gh was installed via MacPorts, nix, or a manual binary, this path does not exist and all three calls will silently fail (`|| true` suppresses errors). The hook will then emit the "Spec traceability added" success message even though nothing was written to the PR.

**Fix:** Resolve `gh` once at the top of the script using `command -v`, falling back gracefully:

```bash
GH=$(command -v gh 2>/dev/null || echo "/opt/homebrew/bin/gh")
if [[ ! -x "$GH" ]]; then
  printf '{"hookSpecificOutput":{"message":"⚠️ pr-traceability: gh CLI not found — traceability skipped."}}'
  exit 0
fi
# Then use $GH everywhere instead of the hardcoded path.
```

### WR-02: uat-gate.sh emits NOT-RUN advisory then continues to check UATG-04 — advisory JSON not well-formed when followed by a second printf

**File:** `hooks/uat-gate.sh:53-67`

**Issue:** When both NOT-RUN rows exist AND the spec-version check fails, the hook prints two separate JSON objects to stdout (lines 56 and 26 inside `emit_block`). Claude's hook runtime expects exactly one JSON object per hook invocation. Two concatenated objects (`{...}{...}`) form invalid JSON, causing the hook runtime to either drop both messages or error.

**Fix:** Collect advisory messages and emit them in a single JSON object. One practical approach: if the version check fails after a NOT-RUN advisory, incorporate the advisory text into the block reason:

```bash
advisory=""
if grep -qE '\| NOT-RUN \|' "$UAT"; then
  not_run_count=$(grep -cE '\| NOT-RUN \|' "$UAT" || true)
  advisory="⚠️ ${not_run_count} NOT-RUN criterion/criteria detected. "
fi

if [[ -n "$uat_version" && -n "$spec_version" && "$uat_version" != "$spec_version" ]]; then
  emit_block "${advisory}UAT GATE: UAT was run against spec v${uat_version} ..."
  exit 0
fi

# Only emit standalone advisory if no block follows
if [[ -n "$advisory" ]]; then
  printf '{"hookSpecificOutput":{"message":"%s"}}' "$advisory"
fi
```

### WR-03: spec-session-record.sh spec-version parsing drops values containing spaces or special characters

**File:** `hooks/spec-session-record.sh:29-30`

**Issue:** `awk '{print $2}'` splits on whitespace, so a spec-version like `1.0 beta` would only yield `1.0`, silently truncating the value. More critically, `tr -d '"'` and `tr -d "'"` do not handle YAML values that contain colons, brackets, or other special characters. While spec-version values are expected to be simple semver strings, an owner typo (e.g., `spec-version: "1.0-rc.1"`) with extra spaces would silently produce a wrong value that causes a UATG-04 mismatch later.

**Fix:** Use a more robust extraction that captures everything after `spec-version: ` and strips only surrounding quotes:

```bash
spec_version=$(grep -m1 '^spec-version:' "$SPEC" \
  | sed 's/^spec-version:[[:space:]]*//' \
  | tr -d '"'"'" \
  | xargs)
```

### WR-04: pr-traceability.sh SPEC.md awk insert targets only the first `<!-- Populated automatically` comment; duplicate sections cause silent double-insertion

**File:** `hooks/pr-traceability.sh:89-92`

**Issue:** The awk script inserts the entry after every line matching `<!-- Populated automatically`. If SPEC.md has two such comments (e.g., a template remnant above the live section), the entry will be duplicated in SPEC.md on each invocation. There is no deduplication guard.

**Fix:** Add a flag to insert only once:

```awk
awk -v entry="$entry" '
  /<!-- Populated automatically/ && !inserted { print; print entry; inserted=1; next }
  { print }
' "$SPEC" > "${SPEC}.tmp" && mv "${SPEC}.tmp" "$SPEC"
```

### WR-05: silver-validate Step 3 AC coverage check uses keyword heuristic without specifying fallback behaviour when AC text contains special regex characters

**File:** `skills/silver-validate/SKILL.md:129-138`

**Issue:** Step 3a instructs the LLM to check whether a task "contains the AC item text (or key noun phrases) in its task name or description" as a fallback when no formal `requirements:` ID link exists. AC text with regex-special characters (parentheses, brackets, pipes) could cause a grep-based match to behave unexpectedly if the skill implementation uses grep. The spec gives no guidance on whether to use literal string matching or regex, nor what to do when the match is ambiguous.

**Fix:** Add a note to Step 3a clarifying the matching strategy:

```
When performing keyword matching, use literal (non-regex) string comparison.
Treat the absence of a formal requirements: link as inconclusive — produce a
WARN (not BLOCK) when keyword heuristic yields a match but no requirement ID
is present. Produce BLOCK only when neither formal link nor keyword match exists.
```

## Info

### IN-01: `_input` variable in spec-session-record.sh is read but never used

**File:** `hooks/spec-session-record.sh:19`

**Issue:** `_input=$(cat)` reads stdin as required by the SessionStart contract, but the variable `_input` is never referenced again. This is intentional (stdin must be consumed) but leaves a dead variable that could confuse future maintainers.

**Fix:** Use the conventional no-op discard pattern to make intent explicit:

```bash
cat > /dev/null  # consume stdin as required by SessionStart hook contract
```

Or add a comment on the existing line: `_input=$(cat)  # must consume stdin; content unused`.

### IN-02: VALIDATION.md closing fence in SKILL.md Step 6 is missing the closing triple-backtick

**File:** `skills/silver-validate/SKILL.md:325`

**Issue:** The markdown code block showing the VALIDATION.md format (opened at line 288) closes at line 325 with a bare backtick fence that ends the non-empty-findings template, but the empty-findings template block beginning at line 328 opens a new fence without closing the outer prose context properly. When rendered, the two code blocks may merge into one in some markdown renderers, making the "If zero findings" branch invisible to the LLM reading the skill.

**Fix:** Ensure the first code block is explicitly closed with ` ``` ` before the "If zero findings, write an empty findings file:" prose line. Review the raw markdown to confirm fence pairing.

### IN-03: Step 17.0 in silver-feature does not specify what to do when SPEC.md has zero acceptance criteria

**File:** `skills/silver-feature/SKILL.md:228-236`

**Issue:** Step 17.0 reads `## Acceptance Criteria` and creates a UAT.md row per criterion. If the section is empty or missing, the UAT.md table would have zero rows but still be written. `uat-gate.sh` would then see a UAT.md with zero FAIL rows and zero NOT-RUN rows and pass the gate — effectively bypassing UAT for features with no formal AC.

**Fix:** Add a guard in Step 17.0:

```
If ## Acceptance Criteria contains zero items, do NOT write UAT.md.
Instead, surface a WARNING to the user:
  "SPEC.md has no Acceptance Criteria. UAT.md cannot be generated.
   Add criteria to SPEC.md before running milestone completion."
Stop and do not invoke gsd-audit-uat.
```
