Create `skills/silver-ingest/SKILL.md` following the exact SKILL.md authoring pattern from silver-spec. The skill has these structural elements:

**Frontmatter** (per silver-feature pattern):
```yaml
---
name: silver-ingest
description: "External artifact ingestion: JIRA/Figma/Google Docs to SPEC.md + DESIGN.md via MCP connectors, plus cross-repo spec fetch with version pinning"
argument-hint: "<JIRA ticket key, --source-url <repo-url>, or artifact URL>"
---
```

**Pre-flight: Load Preferences** — identical pattern to silver-spec (grep §10, display banner). Banner shows:
```
SILVER BULLET > INGEST

Ticket:  {$ARGUMENTS or "(not specified)"}
Mode:    {artifact-ingest | cross-repo-fetch -- detected in Step 0}
```

**Step-Skip Protocol** — identical to silver-spec. Non-skippable gates: `Step 6: Assemble SPEC.md Draft`, `Step 7: Write INGESTION_MANIFEST.md`.

**Step 0: Mode Detection + Prerequisite Check** (per INGT-07 resumability):
- Parse $ARGUMENTS: if starts with `--source-url` -> cross-repo mode (jump to Step 5). Otherwise -> artifact-ingest mode.
- Check if `.planning/INGESTION_MANIFEST.md` exists. If yes, read it and load prior artifact statuses. Display: "Resuming from prior run: {N} artifacts already succeeded."
- Check if `.planning/SPEC.md` exists -> augment mode (read existing spec-version, increment). If not -> greenfield mode.
- Verify MCP connector availability: attempt to list available MCP tools. For each connector (Atlassian, Figma, Google Drive), note availability. Do NOT hard-block if any connector is unavailable -- note as "[CONNECTOR UNAVAILABLE]" and continue.

**Step 1: JIRA Fetch** (conditional -- only if JIRA ticket key provided in $ARGUMENTS, per INGT-01):
- If manifest shows this ticket as `status: success` from prior run, skip with message "JIRA ticket {key} already ingested (from prior run). Skipping."
- Call `jira_get_issue` with the provided key via MCP.
- Extract: summary, description, acceptance criteria field, linked issues, attachment metadata.
- For each Confluence URL found in description body: call `confluence_get_page` to fetch page content (per INGT-02).
- Parse description for Google Drive URLs, Figma URLs, Confluence URLs -- store for Steps 2-4.
- If MCP call fails: record `status: failed` with error reason. Insert `[ARTIFACT MISSING: JIRA fetch failed -- {error}]` for Step 6.
- Record JIRA entry in in-memory artifact list (NOT written to manifest yet -- per Pitfall 6 from research).

**Step 2: Artifact Link Resolution** (conditional -- per INGT-02):
- For each URL found in JIRA description in Step 1:
  - Google Drive URLs -> queue for Step 4
  - Figma URLs -> queue for Step 3
  - Confluence URLs already fetched in Step 1
- If no JIRA ticket was provided but user gave a direct Figma or Google Doc URL in $ARGUMENTS, queue those directly.

**Step 3: Figma Extraction** (conditional -- if Figma URL present, per INGT-03):
- If manifest shows this Figma URL as `status: success` from prior run, skip.
- Prompt user: "Open Figma and select the target frame(s) for extraction, then confirm with Y."
- Wait for user confirmation. Do NOT call MCP tools until confirmed.
- Call `get_design_context` via Figma MCP to get structured design representation.
- Call `get_variable_defs` via Figma MCP to get design tokens (colors, spacing, typography).
- Read `templates/specs/DESIGN.md.template` for canonical structure.
- Write `.planning/DESIGN.md`:
  - Populate `## Screens` from design context (component hierarchy, layout)
  - Populate `## Components` from component names, state variants
  - Populate `## Behavior Specifications` from interaction patterns if available
  - Populate `## Design Tokens` from variable definitions
  - Set frontmatter: `figma-url:` from the URL, `linked-spec: .planning/SPEC.md`, `last-updated:` today
- If MCP call fails or returns empty: record `status: failed`, insert `[ARTIFACT MISSING: Figma extraction failed -- {error}]` in DESIGN.md sections instead of empty sections.
- Record Figma entry in in-memory artifact list.

**Step 4: Google Docs Extraction** (conditional -- if Google Doc URL present, per INGT-04):
- If manifest shows this URL as `status: success` from prior run, skip.
- Attempt extraction via Google Drive MCP `read_document` tool.
- If MCP unavailable, attempt via WebFetch tool as fallback (for publicly accessible docs).
- Extract text content. For documents with embedded images, note: "Image at [position] -- manual review recommended" (vision extraction is best-effort).
- If extraction fails: record `status: failed`, store `[ARTIFACT MISSING: Google Doc extraction failed -- {error}]` for Step 6.
- Record Google Doc entry in in-memory artifact list.

**Step 5: Cross-Repo Fetch** (--source-url mode only, per REPO-01):
- Parse repo URL from $ARGUMENTS (format: `https://github.com/{owner}/{repo}`).
- Fetch main repo's SPEC.md using gh CLI:
  ```bash
  /opt/homebrew/bin/gh api repos/{owner}/{repo}/contents/.planning/SPEC.md --jq '.content' | base64 -d > .planning/SPEC.main.md
  ```
- If gh CLI fails (auth issue), try GitHub raw URL:
  ```bash
  curl -sL "https://raw.githubusercontent.com/{owner}/{repo}/main/.planning/SPEC.md" > .planning/SPEC.main.md
  ```
- Prepend `<!-- READ-ONLY: fetched from {source-url} on {date}. Do not edit. -->` to the file.
- Extract `spec-version:` from fetched file frontmatter. Display: "Fetched SPEC.md (v{version}) from {repo}."
- Record cross-repo fetch in in-memory artifact list.
- Skip to Step 7 (no SPEC.md assembly needed for cross-repo mode).

**Step 6: Assemble SPEC.md Draft** [NON-SKIPPABLE] (per INGT-05, INGT-06):
- Read `templates/specs/SPEC.md.template` for canonical structure.
- Determine spec-version: greenfield = 1, augment = existing + 1.
- Populate sections from collected content:
  - `## Overview` -- from JIRA summary + description
  - `## User Stories` -- derive from JIRA acceptance criteria field if present, else `[ARTIFACT MISSING: no acceptance criteria in JIRA ticket]`
  - `## UX Flows` -- from JIRA description flow references if present
  - `## Acceptance Criteria` -- from JIRA AC field
  - `## Assumptions` -- one `[ASSUMPTION: ...]` block for every gap or ambiguity found during ingestion
  - `## Open Questions` -- from linked issues or unresolved references
  - `## Out of Scope` -- leave as template placeholder unless JIRA explicitly states scope
- For every failed artifact: insert `[ARTIFACT MISSING: {reason}]` in the relevant section. NEVER write an empty section.
- Set frontmatter: `spec-version:`, `status: Draft`, `jira-id:` from ticket key, `figma-url:` if present, `source-artifacts:` list of all URLs, `created:` and `last-updated:` dates.
- Write to `.planning/SPEC.md`.
- Display: "Draft SPEC.md written (v{version}). {N} sections populated, {M} artifacts missing."
- Recommend: "Run /silver:spec to refine this draft through Socratic elicitation."

**Step 7: Write INGESTION_MANIFEST.md** [NON-SKIPPABLE] (per INGT-05, INGT-07):
- Write `.planning/INGESTION_MANIFEST.md` with the manifest schema from research:
  ```markdown
  ---
  run-id: {YYYY-MM-DD-HHmm}
  jira-ticket: {key or null}
  source-url: {url or null}
  last-updated: {ISO timestamp}
  ---

  # Ingestion Manifest

  ## Artifacts

  | Artifact | Type | Status | Error |
  |----------|------|--------|-------|
  | {artifact} | {type} | {success/failed/skipped} | {error or --} |
  ```
- Include ALL artifacts attempted during this run, including those skipped from prior manifest.
- This is the final write -- all artifact statuses are committed atomically at this point.

**Step 8: Commit all artifacts**:
```bash
git add .planning/SPEC.md .planning/DESIGN.md .planning/INGESTION_MANIFEST.md .planning/SPEC.main.md 2>/dev/null || true
git commit -m "spec(ingest): {jira-key or 'cross-repo' or 'manual'} ingestion v{spec-version}"
```

**Step 9: Summary** -- display closing banner:
```
INGESTION COMPLETE

Source:       {JIRA key / source-url / manual}
Artifacts:    {N} attempted, {S} succeeded, {F} failed
Spec version: {spec-version}
DESIGN.md:    {created / updated / skipped}

Next step: run /silver:spec to refine the draft spec through elicitation.
```

If any artifacts failed, add:
```
{F} artifact(s) failed ingestion. Re-run /silver:ingest to retry.
Review .planning/INGESTION_MANIFEST.md for details.
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP connector -> skill | External data from JIRA/Figma/Google enters SB via MCP; skill consumes as-is |
| gh CLI -> SPEC.main.md | Remote repo content fetched into local read-only cache |
| User input -> $ARGUMENTS | JIRA ticket key or repo URL parsed in skill instructions |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-01 | Spoofing | gh CLI auth | accept | gh CLI manages its own OAuth/token auth; SB does not handle credentials. Low risk — spec content is not sensitive. |
| T-13-02 | Tampering | SPEC.main.md | mitigate | READ-ONLY header + silver-bullet.md §0 instruction forbids editing. Version validation at session start detects tampering. |
| T-13-03 | Information Disclosure | MCP connector logs | accept | SB skill does not echo credentials. MCP connectors handle auth lifecycle independently. |
| T-13-04 | Denial of Service | MCP connector timeout | mitigate | Each connector failure produces [ARTIFACT MISSING] and continues. No single connector blocks the entire ingestion run. |
| T-13-05 | Elevation of Privilege | $ARGUMENTS injection | accept | $ARGUMENTS is a natural-language string parsed by the AI, not passed to eval or shell expansion in the skill itself. gh CLI commands use explicit parameter placement, not string interpolation. |
</threat_model>
